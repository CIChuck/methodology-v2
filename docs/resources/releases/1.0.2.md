## Summary

GenDev 1.0.2 makes the distribution boundary a first-class concept. The 1.0.1
release was validated exclusively inside the methodology authority
repository; the first dogfood project and the first cold external assessment
of an installed product repository showed that methodology metadata changes
meaning at the installation boundary, and that 1.0.1 shipped
authority-context tooling into product repositories where its questions are
meaningless. Every finding, its evidence, and its disposition is recorded in
the remediation plan.

## Changes

- Installation is a recorded event: the installer writes
  `docs/methodology/schema/installation.json` with methodology version,
  source remote, tag, commit, install date, installer options, and a sha256
  digest of every installed file (target-owned AGENTS.md excluded).
- The doctor reports context (authority or installed), installed provenance,
  and verifies installed-file integrity, naming modified or missing files and
  exiting 5 on failure.
- Authority-context tools are context-aware: `check-doc-coherence.sh` and
  `check-lifecycle-coherence.py --mode release` refuse installed context with
  exit 4 and are no longer shipped; `test-methodology.sh` restricts product
  context to the applicable suite set and refuses inapplicable requests with
  exit 4.
- Product repositories receive a product-scoped CI workflow (doctor with
  integrity check, methodology coherence, shell syntax, both platforms)
  rendered at install, replacing the authority workflow that failed on first
  push in 1.0.1 installs.
- Provenance pinning is a tool: `scripts/pin-provenance.sh [--check]` repins
  Derived from entries using the identical computation the staleness detector
  uses, and the staleness warning names the tool as its remedy.
- Enforcement legibility: the authoring templates state the provenance pin
  format at the Derived from block, and the architecture template shows the
  labeled design-interrogation answer form the checker requires.
- The practitioner runbook ships at
  `docs/resources/runbook/g0-to-g7-runbook.md`, field-validated through G3.
- Suite hygiene: repo-local scratch is cleaned on all exit paths, and
  installs ship a `.tmp/` ignore rule.
- Installed-context regression suite (IC-001 through IC-008) builds the
  distribution context end to end inside authority CI, so every future
  release validates on both sides of the boundary.

## Prerequisites

Unchanged from 1.0.1: bash 4+, git, python3, ripgrep. Verify with
`./scripts/gendev-doctor.sh`.

## Publication Gates

1. Local validation passes.
2. Remote CI is observed green on the release branch, including the
   installed-context suite.
3. Branch protection remains configured on `master`.
4. Independent review records a non-pending result.
5. Release identity agrees across all active surfaces.
6. Annotated tag `v1.0.2` is created at the reviewed release commit.
7. GitHub Release `GenDev 1.0.2 Distribution Context` is published.

Publication procedure: the final publication commit flips the release index,
release note, schema README, and registry publication status to published
state, the annotated tag is created on that commit immediately, and commit
and tag are pushed together.

## Known Residuals

None accepted at plan time; residuals require an explicit human disposition
recorded in the execution log.
