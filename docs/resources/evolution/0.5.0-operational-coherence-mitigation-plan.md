## 2. Findings

### OM-001: F-001 Through F-020 Lack Required Final Closure Records

Severity: High
Disposition: Fixed in this remediation branch

Evidence:

- The remediation plan's Definition of Done requires a per-finding closure record with root cause,
  authority basis, task IDs, files changed, positive tests, independent negative tests, macOS result,
  Ubuntu result, review references, residual risk, approver/date, and final status.
- The execution log still contains the original F-001 through F-020 ledger with states such as
  `Reproduced`, `Planned`, and `Gap confirmed`.
- Later work-package sections summarize completion, but they do not provide the required final
  closure record for each original finding.

Risk:

The candidate can appear complete without a reader being able to trace every original finding to
specific implementation evidence, negative tests, platform results, residual disposition, and final
status.

Mitigation:

1. Add a final `F-001 Through F-020 Closure Record` appendix to the execution log or a sibling
   closeout artifact.
2. For each finding, record the plan-required fields exactly.
3. Mark final status as `CLOSED`, `RESIDUAL_ACCEPTED`, or `BLOCKED`; do not infer closure from
   work-package status alone.
4. Add a release-coherence check that fails when any F-001 through F-020 row lacks final status,
   macOS result, Ubuntu result or explicit residual, and at least one positive and negative evidence
   reference.

Exit criteria:

- Every F-001 through F-020 row has a complete final closure record.
- `./scripts/test-methodology.sh --suite release-coherence` enforces the closure-record shape.
- The final local matrix is rerun after the closure-record addition.

### OM-002: Remote Ubuntu/macOS CI Remains Unobserved

Severity: High for publication; Medium for local candidate closeout

Evidence:

- The review document lists remote GitHub-hosted Ubuntu/macOS CI as an unobserved residual risk.
- WP-10 is recorded as "Local complete; remote CI pending observation."
- The plan's Definition of Done requires CI to run all registered suites on macOS and Ubuntu and
  fail on missing or skipped suites.

Risk:

The local macOS matrix is strong evidence, but it does not satisfy the full cross-platform release
condition. Publication could proceed with unproven Ubuntu behavior or unobserved GitHub workflow
configuration.

Mitigation:

1. Keep remote CI out of the merge gate until it is configured.
2. Record unconfigured/unobserved remote CI as a known residual/omission in closeout evidence.
3. Before publication, configure remote CI, run it for the exact publication candidate, and capture
   job names, commit SHA, platform, conclusion, and artifact/log links.
4. If publication proceeds without remote CI, record an explicit named waiver.

Exit criteria:

- Merge is not blocked solely by remote CI.
- Publication is blocked until remote Ubuntu and macOS workflow jobs are configured and observed
  green for the exact candidate commit, or the gate is explicitly waived.

  synchronization, delivery-state cleanup, event-template projection, delivered
  reference-graph/documentation-coherence suites, and generated contract refresh.

### OM-004: WP-11 Task Evidence Is Too Compressed

Severity: Medium
Disposition: Fixed for candidate scope; publication-specific historical-tag and release-mode work
remains.

Evidence:

- The remediation plan defines OC-WP11-T001 through OC-WP11-T020, including migration assessment,
  prior-release fixture validation, release identity, historical ratification search, independent
  review, and final matrix rerun.
- The execution log's WP-11 section records release identity notes, review findings, focused
  validation, and final local validation, but it does not include a task-by-task ledger for
  OC-WP11-T001 through OC-WP11-T020.
- WP-10 explicitly says full per-release-shape migration reporting is deferred to WP-11 migration
  artifacts, but WP-11 does not show per-release-shape fixture evidence.

Risk:

Release-prep closeout depends on a small set of summary statements rather than auditable task-level
evidence, especially for migration and historical release handling.

Mitigation:

1. Add a WP-11 task ledger mapping OC-WP11-T001 through OC-WP11-T020 to changed files, commands, and
   outcomes.
2. For migration, either add representative 0.1 through 0.4 fixture evidence or narrow the release
   claim to "migration guide defined; backfill suite tested; historical migration fixtures pending."
3. Link historical ratification search commands and results for 0.2 through 0.4.
4. Verify the exact intended commit for each historical 0.2 through 0.4 tag and prepare
   retrospective tag commands for separate approval.
5. Update the review package to cite the expanded WP-11 evidence.

Exit criteria:

- WP-11 has task-level evidence comparable to WP-02 through WP-10.
- Any migration or historical-tag gap is either tested or explicitly residual.
- Proposed historical tags are backed by exact commit evidence and identified as retrospective.

### ER-002: The Approved Plan's Own Status And Checklist Are Stale After Execution

Severity: Medium

  blocked pending WP-01 completion.
- The execution log records WP-00 through WP-11 candidate-scope completion.
- The plan's Definition of Done checklist remains unchecked.

Risk:

The plan is a useful approved baseline, but readers can confuse its historical authorization state
with current remediation state.

2. Add a short "Execution supersession note" near the top of the plan, or create a sibling closeout

  status.
- No reader has to infer current state from contradictory plan and execution-log endings.

### ER-003: Worktree Hygiene Violates The Plan's Repository Hygiene Definition Of Done

Severity: Medium
Disposition: Initial residue removed in this remediation branch; suite cleanup hardening remains.

Evidence:

- `git diff --check` is clean.
- Before this follow-up, the worktree contained untracked `.tmp/` runtime-test fixtures and
  `scripts/__pycache__/`.
- This follow-up removed those generated local residues.
- The plan's Definition of Done requires test runs to leave no untracked fixture or temporary
  residue.

Risk:

Release-prep evidence can be polluted by local test residue. It also makes it harder to review the
true intended diff.

Mitigation:

1. Confirm the residue is generated and not user-authored evidence. Completed in this branch.
2. Remove untracked generated residue after review approval. Completed in this branch.
3. Add or repair cleanup traps in the suites that created `.tmp/` and Python bytecode, or add
   appropriate ignore rules only for intentionally retained debug artifacts.
4. Add a hygiene check that fails when `.tmp/`, `__pycache__/`, or suite output residue remains
   after a passing aggregate run.

Exit criteria:

- `git status --short --untracked-files=all` shows only intended source artifacts.
- A post-test hygiene check proves no generated residue remains after the full matrix.

### OM-005: Independent Review Record Is Too Thin For The Declared Review Contract

Severity: Medium

Evidence:

- The review package requires findings to name affected authority/code, risk, remediation, and
  required test.
- The review result lists three findings and residual risks, but it does not preserve command
  transcript, exact diff range, file/line evidence, or a complete reviewed-artifact inventory.
- The execution log says the first cold reviewer stopped because files changed during review, but the
  final review package does not preserve that interruption evidence or the exact final review range.

Risk:

The review may be substantively correct, but future readers cannot fully reproduce what was reviewed
or prove reviewer independence and exact-snapshot scope from the review artifact alone.

Mitigation:

1. Add an independent-review transcript appendix with exact reviewed revision, diff range, command
   list, and reviewed artifact inventory.
2. For each finding, add affected file(s), risk, remediation commit or diff reference, and required
   test.
3. Record the interrupted first review as an invalidated attempt, then identify the fresh final
   review snapshot.

Exit criteria:

- The review package is independently auditable without relying on chat/session memory.
