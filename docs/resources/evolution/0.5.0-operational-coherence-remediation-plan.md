## 1. Document Control And Approval Semantics

This document is the comprehensive remediation plan requested after the 2026-07-10 independent
repository review. It converts every confirmed High and Medium finding, plus directly adjacent
defects needed to close those findings honestly, into sequenced implementation work.

The plan is intentionally stored under `docs/resources/evolution/`. That directory is not active
methodology authority. Approval of this plan endorses the proposed direction and authorizes WP-00
plus drafting/validation work in WP-01, but it does not make the plan itself constitutional
authority. WP-01 must render D-001 through D-018 into durable decision records and then stop for a
second, explicit human ratification. Only that ratification authorizes WP-02 through WP-11 and makes
the accepted decisions available as authority for the constitution, guides, templates, schemas, and
tooling changes. This mandatory pause is part of executing the approved plan, not an optional
publication approval.

Approval of this plan does not, by itself, authorize:

- merging to `master`;
- creating or publishing Git tags;
- pushing branches or tags;
- rewriting historical Git objects or historical gate-log events;
- automatically upgrading downstream projects;
- weakening a downstream project's accepted security or governance authority.

Those actions require their own explicit approval when reached. Implementation may prepare the
changes, validation evidence, release notes, and recommended tag commands without performing the
publication step.

## 2. Executive Summary

GenDev has a strong core purpose: replace chat-driven AI implementation with durable authority,
traceability, explicit scope, verification-first planning, human governance, independent review,
controlled remediation, and as-built reconciliation. The review found that the purpose is clearer
than the current operational implementation.

The most serious defects are systemic:

- fresh initialization emits an incomplete construction directive and still reports a clean check;
- the documented gate-closing tool is unusable on stock macOS and does not maintain the full control
  plane even where it runs;
- install and backfill workflows cannot produce the state they promise;
- approval, gate-transition, and staged-change checks admit false negatives;
- `G5.0`, phase exit, G6 review readiness, G7 acceptance, and G9 close-out are not expressed as one
  coherent state machine;
- Verification-First, canonical naming, project identity, task traceability, release identity, and
  current examples have drifted across duplicated documentation surfaces;
- CI tests a narrow regression slice rather than the user-visible workflows the repository ships.

The remediation therefore cannot be a set of isolated text edits. It must proceed in dependency
order:

```text
accepted lifecycle and authority decisions
  -> machine-checkable lifecycle registry and red regression tests
     -> normative lifecycle and artifact reconciliation
        -> canonical templates and task-executable tactical plans
           -> checker and immutable-snapshot enforcement
              -> portable transition tools
                 -> safe install, backfill, hooks, and metrics
                    -> current docs and executable examples
                       -> migration, independent review, and release close-out
```

The target is not merely a green repository. The target is a baseline whose documented happy paths
work end to end on both supported platforms and whose negative tests demonstrate that invalid state
cannot pass silently.

## 3. Purpose And Success Definition

### 3.1 Purpose

This plan will make GenDev operationally consistent with its stated method by:

- defining one unambiguous lifecycle model for gates, phase checkpoints, and phase exits;
- binding every required artifact to an authoring point, approval point, event, and completion test;
- making fresh initialization and phase scaffolding template-driven;
- making gate and phase transitions portable, complete, recoverable, and auditable;
- enforcing approvals and transitions against exact durable events;
- evaluating staged and range checks against immutable Git snapshots;
- making installation and mid-stream adoption non-destructive and self-validating;
- making tactical implementation plans contain stable executable tasks;
- eliminating canonical-path, identity, Verification-First, version, and active-guide drift;
- adding current C1 and C2 examples that demonstrate the current release honestly;
- establishing a cross-platform regression suite that proves both pass and fail behavior;
- publishing an opt-in migration path for existing initialized projects.

### 3.2 Program Success Standard

The program is successful only when all of the following are true:

```text
the lifecycle has one documented and mechanically validated interpretation;
fresh initialization is checker-clean and template-faithful;
phase scaffolding supports every declared phase id without hard-coded Phase 1 assumptions;
no phase exits without required verification, review, reconciliation, and close-out evidence;
no major gate moves without a newly added exact transition event;
no Accepted artifact passes strict mode without durable approval evidence;
staged enforcement is independent of unstaged worktree state;
close-gate and phase-transition tools work on stock macOS and Ubuntu;
install and backfill preserve target-owned files and produce honest validation results;
current templates obey identity, canonical path, provenance, and lifecycle bindings;
tactical task IDs make EC-6 and traceability mechanically meaningful;
README, constitution, registry, project template, release index, and release notes agree on version;
current C1 and C2 examples validate at their declared lifecycle state;
all positive and negative regression suites pass on macOS and Ubuntu;
an independent conformance review reports no unresolved blocking findings;
migration guidance exists and no downstream project is silently upgraded;
release publication remains separately human-approved.
```

## 7. Proposed Decisions For Approval

### D-001: Release Shape

Use one umbrella release, `0.5.0-operational-coherence`, with independently reviewable work
packages. Preserve each baseline version claim exactly as observed during implementation: the
baseline currently reports 0.1 on some active surfaces and 0.4 in the project template. Do not
silently normalize that split to 0.4. Reconcile every active version surface atomically to 0.5 only
in release prep after all required verification and independent review pass. A separately reviewed
0.4 normalization is out of scope unless the approver adds it by amendment.

During WP-01 through WP-10, the new registry identifies itself as a proposed 0.5 candidate and
records the observed active claims separately; it must not claim 0.5 is the released current version.
WP-11 changes candidate/release status and all synchronized active version fields in one reviewed
release-prep change.

### D-002: Platform Contract

Installed shell tooling must run under:

- macOS system Bash 3.2 and BSD `sed`, `awk`, `grep`, `date`, `tar`, and `mktemp` behavior;
- Ubuntu GitHub-hosted runner Bash and GNU utilities.

No code may use `mapfile`, associative arrays, `sed -i`, GNU `0,/regex/` addressing, GNU-only `cpio`
flags, or an undeclared package dependency.

### D-003: Core Dependency Policy

Core installed commands remain dependency-light: Bash 3.2, portable utilities, Git, and tar. A
baseline-only coherence validator may use Python 3 standard library to parse JSON because ad hoc JSON
parsing in shell would be less reliable. If shipped to downstream projects, that validator must be
clearly optional unless the target explicitly enables the baseline-maintenance CI profile.

### D-004: Two-Axis G5 State Model

`project.current_gate` records the active major gate. `phase.phase_position` records the most
recently completed interior checkpoint.

- `G4 -> G5` enters phase planning after governance is accepted.
- `G5.0` means the phase plan is accepted and the phase loop is authorized.
- `G5.0` is a `phase_checkpoint`, not a `gate_transition`.
- `project.current_gate` remains `G5` at `G5.0` and throughout all phase checkpoints.
- `G5.<id>.1` through `.4` update phase state only.
- the sole `G5 -> G6` transition occurs after every declared phase has one valid exit and final
  integration/regression criteria pass or have an approved residual disposition.
- `G8 -> G9` is the terminal major transition. It sets `project.current_gate: G9` and
  `project.status: closed` after final as-built and operational evidence are accepted.
- G9 has no outgoing gate transition. A later change begins through an amendment, regression, or
  explicitly governed new-project/next-product cycle; it never invents G10.

### D-005: Complete Phase Exit

A `G5.<id>.4` phase exit requires pinned evidence for:

1. implementation candidate revision and implementation evidence;
2. test/UAT plan and executed result;
3. independent phase review and reviewed revision;
4. remediation status, including an explicit `not_required` case backed by a clean review;
5. cumulative traceability update;
6. per-phase as-built close-out;
7. phase learnings;
8. regression coverage, residual risks, amendments, and named approval.

G6 remains aggregate whole-build review readiness. G7 remains aggregate acceptance. G9 owns a
separate project-level as-built close-out. A per-phase close-out never substitutes for G9.

### D-006: Phase Exit Approval

Every phase exit requires approval from a named human. C1/C2 may delegate phase-exit approval to a
different named human only when `project.yaml` identifies that person or governed role assignment
and the gate log records who delegated, who accepted the delegation, its scope, and its duration.
Delegation never permits an agent, model, automation identity, or unbound role label to approve
phase exit. C3 requires the specifically authorized named human under the project approval policy.
Major residual findings always require explicit named-human risk acceptance; critical findings
block exit.

### D-007: NAA Authority Boundary

Retain the name "No Undeclared Abstractions," but apply it to authority-bearing abstractions:

- domain/business entities and invariants;
- persisted, shared, externally exchanged, or security-relevant fields;
- public and cross-component interfaces/contracts;
- architectural components and ownership boundaries;
- trust, identity, authorization, audit, and lifecycle boundaries.

Private helpers, local adapters, internal value types, test fixtures, fakes, mocks, and framework
generated types may be introduced by an approved tactical task when they add no product concept,
persisted/shared state, public contract, ownership boundary, or governance behavior. They remain
subject to YAGNI, KISS, DRY, SRP, and Least Astonishment.

### D-008: Tactical Identifier Contract

Use stable identifiers:

```text
Workstream: PH-<phase-id>-WS<NN>
Task:       PH-<phase-id>-T<NNN>
```

Examples: `PH-1-WS01`, `PH-1-T010`, `PH-10-5-T020`.

Phase IDs may contain lowercase/uppercase alphanumerics and internal hyphens. The numeric suffix is
fixed-width and is parsed from the final segment. IDs become immutable when a tactical plan is
Accepted. Retired IDs are marked deferred or superseded and are never reused.

### D-009: Canonical Artifact References

Use `docs/project/design/` as the one canonical location for technique-specific supporting design
artifacts. Do not create a competing `docs/project/supporting/` location.

Revise the reference model to distinguish:

- references to another canonical authority/evidence artifact, such as `tested-by` a canonical test
  plan; and
- references to a technique-specific supporting artifact under `docs/project/design/`.

Each typed reference must declare the target kind so the graph validator can apply the correct path,
identity, authority-direction, and cycle rules.

### D-010: Coverage Policy

Replace the universal phrase `90% meaningful coverage` with a project-declared verification coverage
contract. The contract records metric, target, scope, measurement command, exclusions, owner, and
shortfall disposition. Templates may recommend a default for C2, but may not claim a percentage is
meaningful without defining how it is measured. A shortfall requires named residual risk and human
acceptance at the applicable checkpoint.

### D-011: Approval Evidence Contract

In strict structured mode, an Accepted artifact requires a complete durable event under `## Gate
Records`. `approvals.latest_decision` is a summary and can never replace the event. The event must
bind exact gate/checkpoint, artifact path, reviewed revision/blob/digest, deterministic resulting
blob/digest under one of D-012's three evidence categories, status, approver, date, checked
statement, risk disposition, enforcement context, and next state.

### D-012: Reviewed Revision Contract

Transition tools require a non-placeholder Git revision that resolves and whose artifact blob
matches the exact reviewed pre-closure bytes. The event records the reviewed revision, canonical
path, reviewed Git blob OID, and a portable content digest. Dirty, untracked, mismatched, missing,
and `TBD` evidence is rejected.

Every event evidence item declares exactly one result category:

1. `new_acceptance_status_only` for the single permitted Ready-for-Approval transformation;
2. `complete_report_unchanged` for a Complete evidence report whose reviewed/resulting blobs match;
3. `accepted_authority_unchanged` for previously Accepted authority whose blobs match and whose
   originating approval-event ID binds the same path and blob.

Approval may transform the authority artifact only by replacing the one canonical header
`Status: Ready for Approval` value with `Status: Accepted`. The transformation must preserve every
other byte, including provenance, body content, whitespace, and line-ending convention. Before any
write, the tool renders that deterministic status-only candidate, verifies the exact one-line
semantic diff, and records its resulting Git blob OID/content digest in the event. Artifact path,
substantive content, provenance, encoding, multiple status lines, or any additional difference
blocks the transition. Evidence reports already at `Complete` are never status-rewritten during
approval; their resulting blob/digest must equal their reviewed blob/digest. Previously Accepted
authority reused as evidence is also byte-identical, cites its originating approval event, and has
equal reviewed/resulting blob OIDs and digests. A multi-artifact event binds every evidence item
separately. Manifest and gate-log mutations remain separate candidate files in the same transaction.
This avoids circularly claiming that the yet-to-be-committed approval transition was the revision
the human reviewed.

### D-013: Legacy Compatibility

New 0.5 projects use strict schema-version-2 validation. Existing projects remain pinned until they
opt in. A visible temporary legacy migration mode may warn about incomplete historical records, but
all events created after the mode is enabled must meet the new strict schema. Migration never
fabricates historical facts and never automatically regresses a project.

### D-014: Scaffold Timing

Fresh project initialization creates project-wide authority and canonical directories, not a
hard-coded partially valid Phase 1 implementation set. Add a phase-scaffold command that creates the
complete per-phase artifact set after the accepted phase plan declares the phase ID. A compatibility
`--seed-phase` option may create a complete first-phase set, but must render every file from the
canonical templates.

### D-015: Release And Historical Tags

Create a release index and verify each historical ratification commit where repository evidence can
prove it. Record an unprovable historical mapping as `unknown` with the search performed; do not
create or recommend that historical tag. An unknown historical mapping blocks 0.5 only if a current
authority/version claim depends on it. Correct contradictory release prose with explicit errata. Do
not create or push any historical tag, or the 0.5 tag, until the maintainer separately approves
publication after reviewing the exact verified mapping.

### D-016: Single Lifecycle Registry

Create a machine-readable JSON lifecycle registry for mechanically determinate enumerations and
bindings. The constitution delegates gate enumeration to the canonical lifecycle model; `gates.md`
remains the normative human explanation. The registry must ship with a same-change coherence
validator so it cannot become an unvalidated additional source of drift.

### D-017: Verification-First Scaling

G2 is class-aware:

- C2/C3 PRDs require mechanically recognizable EARS-formed acceptance criteria;
- C1 may use plain, concrete, observable acceptance criteria instead of EARS;
- every class requires explicit unwanted-behavior criteria wherever an error/failure path exists.

G3 requires a human-approved verification specification and proportional design-verification
interrogation for every class. The registry, checker, transition tool, templates, and examples must
encode both the C1 alternative and the C2/C3 EARS rule; C1 is not silently skipped from
Verification-First.

### D-018: Post-Loop Evidence Chain

Use the gate-by-gate G6-G9 model in Section 8.6. In particular:

- G6 owns aggregate implementation/readiness evidence;
- G7 owns final review, aggregate remediation, final verification/UAT, traceability, and acceptance;
- G8 owns deployment readiness, runbook, release scope, operations, rollback, monitoring, the
  disposition-specific value prerequisites defined below, and a separate named-human
  `deployment_approval` event recorded before any production action;
- the terminal G8-to-G9 transition owns deployment or explicit non-deployment disposition,
  post-deployment/operational results, value-review disposition, final traceability, and project-level
  as-built closeout.

Value evidence is never silently dropped. Value artifacts retain the normal artifact `Status`
vocabulary and separately declare `value_review.disposition: complete | not_due | not_applicable`.
The disposition requires evidence, trigger/date/owner, or rationale/approver respectively.
Aggregate remediation and value-review disposition have canonical artifacts and manifest bindings.

## 8. Target Lifecycle And Artifact Model

### 8.1 Major Gate And Phase State

Recommended active-loop manifest shape:

```yaml
project:
  current_gate: G5

approvals:
  current_gate:
    gate: G5
    status: approved

phase:
  loop_schema_version: 2
  loop_status: active
  phase_position: G5.0
  current_phase_id: null
  phase_plan: docs/project/build-plan/phase-plan.md
  latest_decision:
    event_type: phase_checkpoint
    position: G5.0
    decision: accepted
    decided_by: "[name]"
    decided_on: YYYY-MM-DD
    record: docs/project/approvals/gate-log.md
  phases:
    - id: "1"
      name: "[Phase Name]"
      status: pending
```

Allowed loop statuses:

```text
not_started
active
blocked
ready_for_g6
closed
legacy_unreconciled
```

Allowed phase statuses:

```text
pending
planning
authorized
in_progress
ready_for_exit
exited
blocked
superseded
```

### 8.2 Checkpoint Bindings

### 8.3 Per-Phase Canonical Artifact Set

For phase `<id>`:

### 8.5 G6, G7, G8, And G9 Boundaries

- G6 is whole-build review readiness. During G6, the aggregate implementation summary, candidate,
  changed-file inventory, cross-phase integration/regression results, deviations, and review package
  are completed. Closing G6 enters G7 for final review; it does not claim final review already ran.
- G7 is final whole-build conformance review, remediation, final verification/UAT, traceability, and
  named-human implementation acceptance. Closing G7 enters G8 with deployment readiness pending.
- G8 is deployment readiness and operations. It owns release scope, environment/config/secrets,
  migration, rollback/irreversibility, monitoring, operational ownership, runbook, and a distinct
  named-human deployment approval before production action.
- G9 is project-level as-built closure after deployment or explicit non-deployment disposition. It
  reconciles actual system state, aggregate traceability, operations, limitations, and future work.
- G8-to-G9 is the terminal transition. It retains the complete G8 operational record, sets
  `project.current_gate: G9` and `project.status: closed`, clears the active role to the registry's
  `none` terminal value, and declares no next gate or mandatory authority artifact.
- A direct `close-gate.sh G9` invocation is invalid because G9 is already closed. Continued product
  work requires the applicable amendment/regression or newly governed project/phase path.

### 8.6 G6-G9 Evidence And Transition Map

This table is normative input to D-018's decision record and the lifecycle registry. Artifact paths
are canonical; a combined C1/C2 form may combine documents but cannot omit any evidence row.

| Transition | Required canonical source evidence and status | Event and state mutation | Human approval | Command |
| --- | --- | --- | --- | --- |
| `G5 -> G6` | All declared `G5.<id>.4` events; exact whole-build candidate revision; cumulative phase-loop integration/regression result; current traceability; enumerated residuals | `gate_transition`; set gate G6, project active, target approval pending, role `implementation-agent`, phase loop closed while retaining final phase position | No additional approval by default because every phase exit is human-approved; a named human must accept skipped critical verification or otherwise nonwaivable residuals | `close-gate.sh G5` |
| `G6 -> G7` | `docs/project/build-plan/implementation-summary.md` Complete with exact candidate, changed files, tests/commands, deviations, integration/regression results, residuals, and final-review package; current traceability | `gate_transition`; set gate G7, project active, target approval pending, role `code-review-agent` | No implementation-acceptance decision yet; named human is required only for an allowed skipped critical verification/risk disposition | `close-gate.sh G6` |
| `G7 -> G8` | `docs/project/review/code-review.md` Complete at exact candidate revision; `docs/project/review/remediation.md` Complete or explicit `not_required`; `docs/project/testing/final-test-uat-report.md` Complete; current traceability; residual-risk disposition; previously Accepted upstream authority bound byte-identically | `gate_transition`; set gate G8, project active, target approval pending, role `deployment-readiness-agent`; bind exact accepted release candidate and deployment/non-deployment intent | Named human accepts the implementation; every critical finding closed; every major residual explicitly accepted with rationale | `close-gate.sh G7` |
| G8 deployment authorization (no gate change) | `docs/project/deployment/deployment-readiness.md` Ready for Approval; `docs/project/deployment/production-runbook.md` Complete; release scope, config/secrets, migration/rollback, monitoring, incident ownership, post-deployment validation, and the disposition-specific value prerequisites below | `deployment_approval`; retain gate G8 and project active; change readiness artifact status only to Accepted; set current G8 approval approved; bind exact release candidate; perform no deployment automatically | Named human deployment/release approver; security approver also required where governance says so | `record-deployment-approval.sh` |
| `G8 -> G9` | Prior valid `deployment_approval` or approved non-deployment disposition; `docs/project/deployment/deployment-record.md` Complete with results/disposition; rollback/incident/monitoring status; `docs/project/as-built/value-review.md` Status Complete with valid `value_review.disposition`; final traceability; `docs/project/as-built/as-built-closeout.md` Complete | terminal `gate_transition` with `terminal_closeout: true`; set gate G9, project status closed, approval summary final, active role `none`, no next gate/role/artifact | Named human project closeout approver; named operational/deployment owner confirms production disposition where deployed | `close-gate.sh G8` |

Value-review disposition rules (separate from artifact `Status`):

- `complete`: all due criteria cite evidence and report `met`, `missed`, or `unmeasurable`; missed or
  unmeasurable outcomes have a follow-up decision;
- `not_due`: record the trigger, expected date when knowable, named owner, evidence source, and next
  review mechanism;
- `not_applicable`: record why the release/project has no value-bearing outcome and the named human
  who accepted that disposition;
- the aggregate value-review artifact itself is `Status: Complete` once one valid disposition is
  fully recorded;
- absence, `TBD`, an unowned future trigger, conflating disposition with artifact status, or treating
  `unmeasurable` as success blocks G8-to-G9.

The registry must also define the exact status transition for each artifact, required reviewed blob
bindings, event criterion IDs, combined-gate representation, and allowed non-deployment path.

  originating event ID when the category is `accepted_authority_unchanged`;

**Strict approval rules:**

For each Accepted gate artifact, require an event that:

- uses the expected gate or checkpoint and approved/accepted decision;
- cites the exact canonical artifact path;
- cites a non-placeholder reviewed revision;

**Exact transition comparison:**

When `project.current_gate` differs between base and candidate:

1. Extract base manifest and gate log from the specified Git base.
2. Evaluate candidate manifest and gate log from the selected snapshot.
3. Require the gate-log path in the candidate change set.
4. Require one new event with exact base `from_gate` and candidate `to_gate`.
5. Require complete approval/event fields and an event ID absent from base.
6. Require candidate manifest current-gate, approval block, active role, phase state, and latest
   decision to agree with the event.
7. For an artifact approval, verify that the reviewed blob matches the exact pre-closure file and

   evidence report result blobs to remain identical.
8. For previously Accepted authority reused as evidence, require byte-identical reviewed/resulting
   blobs and a valid originating approval-event ID for the same path/blob.
9. Reject missing, old, wrong-direction, duplicate, rejected, or edited historical events.

## 12. Migration And Compatibility Strategy

Migration is an explicit product surface of this remediation, not a documentation afterthought.
The migration path must preserve historical truth, leave project-owned content intact, and avoid
claiming conformance until the target project has actually satisfied the 0.5 contract.

### 12.1 Migration Principles

Every migration implementation and operator procedure must follow these rules:

1. **Assess before mutating.** The first command is always a read-only assessment that emits a
   machine-readable and human-readable report.
2. **Preserve project content.** Existing authority, evidence, approval, source, test, and custom
   hook bytes are never overwritten merely because a current template differs.
3. **Preserve history.** Historical gate-log events remain append-only. Corrections are new
   reconciliation or supersession events.
4. **Do not infer missing facts.** A missing approver, date, reviewed revision, result, or risk
   disposition stays unknown until a human supplies defensible evidence.
5. **Keep version changes last.** `methodology_version` and schema version are updated only after
   the migrated candidate validates and a named human approves adoption.
6. **Separate filesystem readiness from lifecycle conformance.** Creating missing directories or
   templates does not imply historical gates were valid.
7. **Make every mutation reviewable.** Before applying changes, show the exact files to create,
   append, or modify and the reason for each action.
8. **Make every mutation recoverable.** Use a project-local transaction workspace, preserve
   preimages, and support deterministic rollback before commit.
9. **Honor pinned releases.** A project can remain on 0.1-0.4; the current baseline must not silently
   reinterpret that project's historical state under 0.5 rules.
10. **Apply strictness prospectively.** Once a project opts into schema version 2, every new event
    must use the strict event contract even when older events remain incomplete.

### 12.2 Project Cohorts

The migration assessor must classify a target into exactly one primary cohort and zero or more
qualifiers. An ambiguous target blocks automatic application and requires human disposition.

| Cohort | Detection summary | Default action | Required approval |
| --- | --- | --- | --- |
| M0 - New clone | No active `docs/project/project.yaml`; no imported authority | Run current installer/init path | Normal initialization confirmation |
| M1 - Installed but uninitialized | Methodology assets present; active project absent | Validate distribution, then initialize | Initialization confirmation |
| M2 - Imported authority, no control plane | Project docs exist; manifest or gate log absent | Non-destructive backfill | Backfill plan approval |
| M3 - Initialized, pre-G5 | Valid manifest/log; major gate G0-G4 | Migrate schema and reconcile current authority | Migration approval |
| M4 - Active G5 phase loop | Major gate G5; incomplete declared phases | Choose pinned completion or explicit mid-phase amendment | Migration plus amendment approval |
| M5 - Post-build, pre-closeout | G6-G8 | Preserve completed phase evidence and reconcile aggregate gates | Migration approval and residual-risk review |
| M6 - Closed project | G9 or durable project close-out | Prefer historical pin; migrate only for continued work | Reopen/amendment approval where applicable |
| M7 - Customized baseline | Forked scripts/templates/schema or local enforcement changes | Produce divergence report; no automatic overwrite | Maintainer disposition per divergence |

Qualifiers include:

- legacy example/reference material accidentally copied as active authority;
- unknown or unparseable version;
- dirty working tree;
- untracked project artifacts;
- custom `core.hooksPath` or pre-existing hooks;
- detached HEAD, shallow history, or missing candidate revisions;
- active amendment or stale-authority markers;
- local canonical paths that conflict with 0.5 paths;
- symlinks crossing the target root;
- incomplete or malformed YAML/Markdown event structures;
- mixed line endings or filenames containing spaces;
- enforcement class or blast-radius class absent or invalid.

### 12.3 Migration Command Contract

The exact command names may be ratified in WP-01, but the operator contract is fixed:

```text
assess -> review report -> approve -> prepare transaction -> apply candidate
       -> validate candidate -> show diff -> approve adoption -> finalize version
```

The implementation should expose this through `backfill-methodology.sh` plus a dedicated migration
mode, or through a narrowly scoped migration script if that yields clearer ownership. Whichever
interface is chosen must support:

```text
--target <repository>
--assess-only
--report <path>
--apply <accepted-report-id>
--non-interactive
--methodology-version <source-version>
--target-version 0.5.0-operational-coherence
--legacy-mode <warn|strict>
--keep-transaction-on-failure
```

Required behavior:

- assessment is the default and has no target mutation;
- `--apply` requires the digest of the accepted assessment report;
- a changed target invalidates the report digest and forces reassessment;
- interactive prompts are forbidden in `--non-interactive` mode;
- unknown options and missing values fail with usage text and nonzero status;
- target paths are canonicalized and constrained to the intended repository;
- a lock prevents concurrent init/backfill/migration/transition operations;
- stdout carries concise operator status; stderr carries diagnostics;
- exit status distinguishes success, nonconformance, invalid input, conflict, and internal failure;
- secrets and environment values are not copied into reports.

### 12.4 Read-Only Assessment

Assessment must capture enough information to plan migration without claiming authority:

1. identify the Git root, current branch, HEAD, dirty state, staged state, and untracked project
   paths;
2. locate methodology assets, active authority, gate log, manifest, schema, phase plan, and current
   directive;
3. detect declared and inferred methodology versions separately;
4. parse the current major gate and phase position without modifying either;
5. inventory canonical, legacy, floating, missing, duplicate, and project-specific paths;
6. validate project identity across structured active artifacts;
7. enumerate historical gate/checkpoint events and report incomplete fields without rewriting them;
8. enumerate phase declarations and evidence classes by phase;
9. locate tactical workstream/task identifiers and unresolved dependencies;
10. identify current coverage policy, NAA wording, enforcement class, and blast-radius class;
11. inspect hook configuration and existing managed/unmanaged hooks;
12. compute byte digests for every proposed-to-change project file;
13. classify each proposed action as `create`, `append`, `transform`, `manual`, or `blocked`;
14. emit a deterministic report ID over target HEAD, relevant file digests, source version, target
    version, and proposed actions.

The assessor must never treat example material as active authority. It must report when an active
artifact appears to contain shipped example identity or placeholder evidence.

### 12.5 Migration Action Classes

| Action class | Allowed automatically after report approval | Prohibited automatic behavior |
| --- | --- | --- |
| Directory creation | Create missing canonical empty directories | Rename or delete non-empty legacy directories |
| Scaffold creation | Add a missing artifact from a current template with explicit `Draft`/pending state | Replace an existing authority file |
| Manifest addition | Create a control-plane manifest from supplied project facts and discovered state | Infer accepted gates or approvals |
| Gate-log addition | Create a log with initialization and migration events | Synthesize historical approvals |
| Event reconciliation | Append a schema-versioned mapping/supersession event | Edit old event text in place |
| Path migration | Add canonical copy or explicit redirect/supersession record after content comparison | Silently move unresolved or conflicting authority |
| Task migration | Add stable IDs with a preserved legacy-ID mapping | Reorder work or invent completion evidence |
| Coverage migration | Add a project-declared policy marked pending until approved | Assume the former 90 percent phrase is measured evidence |
| NAA migration | Pin old directives or amend/reissue under accepted 0.5 language | Retroactively declare old private helpers nonconforming |
| Hook migration | Install a managed dispatcher while preserving existing behavior | Overwrite or bypass an unmanaged hook |
| Version finalization | Update synchronized version fields after green validation and approval | Update version before candidate validation |

### 12.6 Mid-Phase Migration

An M4 project must choose one of two explicit paths:

**Path A: complete the active phase under the pinned release.**

- retain the existing directive, tactical plan, gate semantics, and checker profile for that phase;
- record that the project intends to adopt 0.5 at the next clean checkpoint;
- complete and close the phase under the pinned release without mixing 0.5 evidence claims;
- assess migration again after closure;
- adopt 0.5 before planning or authorizing the next phase.

**Path B: amend and reissue the active phase under 0.5.**

- open an amendment identifying which accepted authority is invalidated;
- classify completed implementation and evidence as reusable, stale, or requiring rerun;
- migrate the phase artifact set and tactical identifiers;
- reissue the construction directive against the accepted amended authority;
- rerun affected verification and review at an exact candidate revision;
- close the amendment only after the 0.5 checkpoint contract is satisfied.

The tooling may recommend a path from state, but it must not choose on behalf of the human. A C3
project always requires an explicit named human decision.

### 12.7 Historical Event Reconciliation

The target schema must define a reconciliation event that can state:

```text
event_type: migration_reconciliation
schema_version: 2
source_methodology_version: <detected or declared>
target_methodology_version: 0.5.0-operational-coherence
historical_event_reference: <stable event identifier or source line anchor>
mapped_gate_or_checkpoint: <target identifier>
mapped_evidence_classes: <explicit list>
unresolved_fields: <explicit list>
provenance: <artifact paths and revisions>
decision: mapped | partially_mapped | not_mapped
approver: <named human when required>
approval_date: <ISO date>
risk_disposition: <explicit>
```

Rules:

- reconciliation does not convert an incomplete old event into a complete strict event;
- unresolved fields remain visible in all subsequent readiness checks where relevant;
- line numbers alone are not durable identifiers after append operations; the schema must provide a
  stable event ID or digest;
- duplicate mappings are rejected unless the later event explicitly supersedes the earlier one;
- evidence is bound to the revision that was actually inspected, not current worktree content;
- a human can accept residual historical uncertainty only where the lifecycle and risk class allow
  it; critical security/approval uncertainty is not waivable by migration convenience.

### 12.8 Migration Transaction And Backout

Application must use a two-phase filesystem transaction:

1. acquire a project-local lock;
2. verify the accepted report digest and all recorded preimage digests;
3. create a transaction directory under Git-ignored project-local state or a secure temporary
   directory on the same filesystem;
4. copy all preimages and record modes, symlink targets, and absence markers;
5. render every candidate output into the transaction directory;
6. run structural, identity, event, lifecycle, and shell validation against the candidate tree;
7. display the candidate change manifest and content diff;
8. after apply approval, atomically replace or create one file at a time while recording progress;
9. on any error or signal, restore applied preimages in reverse order;
10. validate restored digests and report rollback status;
11. retain a recovery manifest when automatic restoration cannot be proven;
12. release the lock only after success or verified restoration.

The migration implementation must never use `rm -rf docs/project`, broad template overlays, or a
force option that bypasses preservation checks.

### 12.9 Migration Acceptance Scenarios

The fixture matrix must include at least:

- clean 0.1, 0.2, 0.3, and 0.4 representative projects;
- an imported-doc M2 project with no manifest/log;
- a pre-G5 project with complete and incomplete historical approvals;
- a G5 project at G5.0 and at each interior checkpoint;
- a multiphase project with one closed and one active phase;
- a G6-G8 project with aggregate evidence;
- a G9 historical project;
- customized templates and scripts;
- conflicting canonical and legacy path content;
- valid existing hooks plus custom `core.hooksPath`;
- dirty, staged, untracked, detached, and shallow Git states;
- malformed version, project identity, event, phase, task, and coverage records;
- injected failure before candidate validation, during first write, mid-apply, after final write, and
  during rollback.

For every fixture, assert:

- assessment mutation count is zero;
- report contents and exit status are deterministic;
- apply refuses stale reports;
- preserved files retain byte digests unless the accepted action names them;
- historical events retain byte content;
- created files have honest Draft/pending state;
- validation cannot claim full conformance when facts remain unresolved;
- rollback restores the exact preimage after each injected failure;
- a successful migration is idempotent;
- a second assessment reports no unplanned migration action.

- control characters, colon/hash/quote/newline values, and Unicode edge cases in human-supplied text;
- duplicate Markdown headings or an event under a noncanonical section.
- deployment approval that changes current gate, lacks readiness/runbook/value-trigger evidence, or
  claims/executes a production action;

Every invalid case must fail for the intended reason. Tests must assert a stable diagnostic code in
addition to human-readable text so wording changes do not erase behavioral specificity.

### 13.7 Immutable Git-State Tests

The guard matrix must construct commits, indexes, and worktrees deliberately:

- clean commit range;
- staged manifest/log/artifact change with no worktree divergence;
- same staged tree plus contradictory unstaged manifest;
- same staged tree plus contradictory unstaged log;
- same staged tree plus contradictory unstaged authority artifact;
- staged deletion with an unstaged recreation;
- staged rename with worktree content change;
- untracked file at a canonical path;
- partially staged file;
- merge commit and multi-commit range;
- path containing spaces;
- symlink inside the evaluated tree;
- missing base, invalid revision, shallow boundary, and empty range.

The implementation must materialize one immutable snapshot from the index or target tree and run all
checks against that snapshot. A test passes only when ambient worktree mutations cannot change the
result for an identical evaluated tree.

### 13.8 Distribution Verification

Distribution tests must build the install set from `scripts/lib/distribution-manifest.txt`, then:

1. archive or copy only declared distributable files into a scratch source;
2. run the real installer into an empty initialized Git repository;
3. compare every installed file, executable bit, and required directory to the manifest;
4. verify no development-only, judge-only, historical, secret, or local state file leaked;
5. initialize a named project through the installed path;
6. run the installed checker and guard help/contracts;
7. scaffold a declared phase;
8. execute representative gate/checkpoint transitions;
9. record a G8 deployment approval and prove it neither changes gate nor invokes production action;
10. install and exercise hooks with both default and custom hooks paths;
11. run installed documentation commands exactly as written;
12. update an already installed target and prove project-owned files are preserved;
13. uninstall managed hook components and prove unmanaged hook behavior remains.

An installed path is not considered valid merely because the source repository passes its own
checker.

### 13.9 Documentation And Example Verification

The active-doc sweep must distinguish:

- normative methodology authority;
- active practitioner guidance;
- templates and project scaffold;
- role and dev-skill operational instructions;
- current runnable examples;
- explicitly preserved legacy/historical records;
- evolution plans and release records.

Checks must flag stale canonical paths, old version claims, old G5 semantics, old NAA wording,
undefined coverage wording, obsolete commands, missing project identity, and references to retired
artifacts in every active category. Historical records are exempt from rewriting but must be
clearly labeled and must not be linked as current authority.

Current C1 and C2 examples must be initialized repositories or complete fixture trees that:

- declare project identity, enforcement class, blast-radius class, methodology version, and schema;
- contain the correct lifecycle artifacts for their documented state;
- pass checker/registry/path/reference validation;
- demonstrate stable tactical IDs and the phase exit evidence contract;
- include exact runnable commands and expected statuses;
- contain no product implementation claim without matching local evidence;
- never become fallback authority for the active project.


## 15. Rollback, Recovery, And Preservation Plan

Rollback has two distinct meanings in this program:

- **tool-level recovery:** restore a target repository after a command fails before its logical
  operation completes; and
- **program-level reversal:** revert an accepted remediation change through normal Git history after
  it is committed.

Neither meaning permits deleting unrelated user work, rewriting published history, or editing an
append-only approval record in place.

### 15.1 Shared Transaction Requirements

All multi-file state-changing commands must use the shared transaction library planned in WP-06.
The library contract must include:

```text
begin_transaction <operation> <target-root>
register_preimage <path>
register_absence <path>
render_candidate <path> <candidate-path>
validate_candidate
apply_candidate <path> <candidate-path>
record_postimage <path>
rollback_transaction
commit_transaction
```

The names are illustrative; the behaviors are mandatory. Implementation must:

- reject a target outside the canonical Git root unless the command explicitly operates on an
  install destination;
- reject symlink traversal that escapes the target;
- create temporary material with restrictive permissions and predictable cleanup;
- prefer a same-filesystem temporary directory so atomic rename is available;
- preserve file mode and distinguish absent paths from empty files;
- fsync or otherwise document durability limitations where relevant;
- write a transaction manifest before the first target mutation;
- install signal traps for `HUP`, `INT`, `TERM`, and normal error paths;
- restore in reverse application order;
- compare restored digests to preimages;
- never print `success` until post-validation and transaction commit complete;
- retain exact recovery instructions when automatic rollback is incomplete.

### 15.2 Command-Specific Recovery

| Command area | Atomic unit | Recovery expectation |
| --- | --- | --- |
| Fresh init | Entire new `docs/project` scaffold plus managed config | Remove only paths proven absent before init; restore any explicitly updated config |
| Phase scaffold | Complete declared phase artifact set | Remove only newly created phase files; never remove pre-existing phase content |
| Gate/checkpoint/phase transition | Gate log, manifest state, approval summaries, role/phase state | Restore every file to exact pre-transition bytes; no orphan event or partial gate advancement |
| Installer/update | Methodology runtime/docs/workflow set and install manifest | Restore overwritten managed files; leave project-owned/unmanaged files untouched |
| Backfill/migration | Accepted action manifest | Restore all transformed files; remove only transaction-created files; preserve imported docs |
| Hook install | Managed dispatcher, fragments, install record | Restore original hook bytes/mode/path behavior exactly |
| Hook uninstall | Managed components only | Recreate managed components on failed uninstall; never remove unmanaged hook content |
| Release identity update | All synchronized active version surfaces | Revert candidate change as a unit before commit; no partially updated version claim |

### 15.3 Program-Level Reversal

Once changes are committed, reversal uses new Git commits. The implementation must not use hard
reset, force-push, history rewriting, or deletion of durable gate-log events. Reversal procedure:

1. identify the exact accepted remediation commit range;
2. classify whether any downstream project has consumed the changed release;
3. stop publication if the release is not yet published;
4. if unpublished, prepare reviewable revert commits and rerun the pre-0.5 plus relevant current
   regression matrix;
5. if published, issue a new corrective release or documented withdrawal rather than moving the
   published tag;
6. append supersession/erratum records for affected release or decision documents;
7. preserve migration reports and evidence as historical records;
8. obtain the same class of human approval required for the affected authority and publication.

Historical annotated tags, once pushed, are immutable for this plan. A wrong tag mapping requires a
new corrective tag/release record and explicit explanation, not silent retagging.

### 15.4 Recovery Manifest

When automatic rollback cannot be proven, the tool exits nonzero and leaves a recovery manifest
containing:

```text
transaction ID and operation
target root and repository HEAD
start time and failure time
applied paths in order
preimage backup path, digest, mode, and original existence
candidate/postimage digest where available
rollback actions attempted and their statuses
paths whose restoration is unverified
exact safe manual commands or procedure
lock state and release procedure
supporting diagnostic log path
```

The manifest must not contain secrets. A subsequent invocation detects the unfinished transaction
and refuses new mutation until the operator recovers or explicitly archives the verified-complete
transaction.

### 15.5 Preservation Proof

Preservation claims close only with byte-level and behavior-level evidence:

- SHA-256 or an available portable digest of project-owned pre/post files;
- file mode and symlink-target comparison;
- Git status before/after;
- hook invocation counters demonstrating existing behavior runs exactly once;
- historical gate-log prefix comparison plus appended-event inspection;
- absence markers proving rollback removed only transaction-created files;
- repeat-run idempotency;
- post-recovery checker and Git integrity checks.

When a platform lacks one digest command, the test harness must choose a declared portable fallback
without silently skipping the assertion.
