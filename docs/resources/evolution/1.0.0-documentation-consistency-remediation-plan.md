# GenDev 1.0 Documentation Consistency Remediation Plan

Status: Executed on `release/1.0.1-documentation-consistency`
Date: 2026-07-11
Branch reviewed: `release/1.0.0-production`
Remediation branch: `release/1.0.1-documentation-consistency`
Target release: `1.0.0`

## Purpose

This plan records the documentation inconsistencies, omissions, and accuracy risks found during a
repository-wide 1.0 consistency pass, with a deeper pass through the Practitioner Guide.

The goal is to make the active documentation precise, complete, and coherent for an AI agentic
developer or human practitioner using GenDev 1.0 in production.

## Scope

In scope:

- Active methodology documentation under `docs/methodology/`.
- Active Practitioner Guide documentation under `docs/resources/practitioner-guide/`.
- Active release, adoption, and evolution documentation under `docs/resources/releases/` and
  `docs/resources/evolution/`.
- Repository-level operator guidance in `README.md` and `AGENTS.md`.
- Automated coherence checks that should protect this documentation from drifting again.

Out of scope unless they confuse active usage:

- Historical release records.
- Provenance snapshots.
- Superseded decision records that are explicitly marked as historical.

## Validation Baseline

The following checks were clean before this remediation plan was created:

```text
./scripts/check-doc-coherence.sh
./scripts/check-lifecycle-coherence.py --mode release
./scripts/test-methodology.sh --suite release-coherence
```

This is useful evidence, but it also exposes a coverage gap: the current automated checks did not
detect several active-documentation inconsistencies listed below.

## Findings

### DC-001: Release State Is Internally Contradictory

Severity: High

Evidence:

- `docs/resources/releases/README.md` says `Latest release: 1.0.0`.
- `docs/resources/releases/README.md` also lists `1.0.0` as `Release branch active` with planned
  tag `v1.0.0`.
- `docs/resources/releases/1.0.0.md` says publication is pending required gates and the planned tag
  is `v1.0.0`.
- `docs/methodology/schema/lifecycle.json` records `registry.status: released`,
  `versions.candidate_status: released`, and `versions.release_stage: production`.
- `docs/methodology/schema/README.md` says `Status: Released production registry`.
- `git tag --points-at HEAD` and `git tag --list 'v1*'` did not show a `v1.0.0` tag.

Impact:

Agents and practitioners can incorrectly treat the release branch as already published. This weakens
publication discipline and makes release gates ambiguous.

Remediation:

- Split published state from pre-publication state in active docs.
- Until publication is actually complete, use wording such as:
  - `Latest published release: 0.5.0`
  - `Active release candidate: 1.0.0`
  - `Planned production tag: v1.0.0`
- Keep lifecycle metadata in candidate or pre-publication form until the publication gates are closed.
- Flip the lifecycle registry and release index to released production only in the final publication
  commit, immediately before tagging.

### DC-002: Active Methodology Role and Dev-Skill Docs Still Use 0.5 Headings

Severity: High

Evidence:

The following active files still contain `0.5 Operational Coherence` headings:

- `docs/methodology/dev-skills/ai-construction-directive-builder.md`
- `docs/methodology/dev-skills/remediation-closeout.md`
- `docs/methodology/dev-skills/tactical-implementation-planner.md`
- `docs/methodology/dev-skills/phase-build-planner.md`
- `docs/methodology/dev-skills/code-conformance-reviewer.md`
- `docs/methodology/dev-skills/traceability-matrix.md`
- `docs/methodology/agents/roles/test-uat-agent.md`
- `docs/methodology/agents/roles/implementation-agent.md`
- `docs/methodology/agents/roles/remediation-agent.md`
- `docs/methodology/agents/roles/phase-planning-agent.md`
- `docs/methodology/agents/roles/as-built-closeout-agent.md`
- `docs/methodology/agents/roles/code-review-agent.md`
- `docs/methodology/agents/roles/deployment-readiness-agent.md`

Impact:

These are active practitioner-facing and agent-facing instructions. Keeping 0.5 headings in the 1.0
release creates avoidable uncertainty about which release owns the current operating model.

Remediation:

- Rename active headings to either `1.0 Operational Coherence Requirements` or release-neutral
  `Operational Coherence Requirements`.
- Prefer release-neutral headings where the content is now stable across releases.
- Add a coherence check that scans active role and dev-skill docs for stale release-scoped headings.

### DC-003: First-Run Preflight Does Not Include 1.0 Operator Commands

Severity: High

Evidence:

- `docs/resources/practitioner-guide/04-starting-a-new-project.md` contains a `First-Run Preflight`
  section, but it does not instruct the practitioner or agent to run `./scripts/gendev-doctor.sh` or
  `./scripts/project-state.sh`.
- `docs/methodology/guides/start-and-next-step-protocol.md` describes first-run preflight but also
  omits those commands.

Impact:

The 1.0 release added operator tooling that makes the methodology more valuable to agentic
developers. If the preflight guidance does not require it, users can start a project without the
most important machine-readable health and state checks.

Remediation:

- Update the Practitioner Guide first-run preflight to include:
  - `./scripts/gendev-doctor.sh`
  - `./scripts/project-state.sh`
  - `./scripts/check-doc-coherence.sh`
- Update the start and next-step protocol with the same command sequence.
- Explain what each command proves without turning the guide into a script reference.
- Require agent prompts to cite or summarize the resulting state before beginning substantive work.

### DC-004: Practitioner Repository Map Does Not Match the 1.0 Scaffold

Severity: Medium

Evidence:

- `docs/resources/practitioner-guide/03-repository-map.md` lists a typical initialized structure
  that omits active 1.0 scaffold directories such as `design/`, `review/`, and `deployment/`.
- The same section lists `docs/project/README.md`, but the current initializer and template do not
  create that file.
- The artifact directory reference omits current lifecycle directories such as `approvals/`,
  `design/`, `review/`, and `deployment/`.

Impact:

The Practitioner Guide can send a user looking for paths that do not exist and can fail to introduce
paths that are now first-class in 1.0.

Remediation:

- Reconcile the repository map against `scripts/init-project.sh` and `docs/project-template/`.
- Remove nonexistent files from the typical tree, or add them to the scaffold if they are intended.
- Add every current first-class artifact directory to the directory map with concise purpose text.
- Add a check that compares the guide's documented scaffold paths to the template paths.

### DC-005: Practitioner Checklists Omit 1.0 Health, State, and Coherence Checks

Severity: Medium

Evidence:

- `docs/resources/practitioner-guide/16-checklists.md` does not include
  `./scripts/gendev-doctor.sh`, `./scripts/project-state.sh`, or
  `./scripts/check-doc-coherence.sh` in the new-project checklist.
- The first agent session checklist asks the agent to identify state, but it does not require the
  1.0 state-reporting command.
- Additional operational checks omit late-gate use of `./scripts/new-artifact.sh`.

Impact:

The checklists are likely to be the fastest reference path for users and agents. If they omit the
1.0 tools, the release value is underexposed and operational discipline depends on longer-form guide
chapters.

Remediation:

- Add the 1.0 operator commands to new-project, first-agent-session, and phase/gate checklists.
- Add `./scripts/new-artifact.sh` to artifact-creation checklist steps where template conformance
  matters.
- Add a checklist item requiring doc coherence validation before publication or release-note edits.

### DC-006: Adoption and Backfill Terminology Conflicts With the No-Legacy-Project Decision

Severity: Medium

Evidence:

- `docs/resources/practitioner-guide/17-common-failure-modes.md` contains `Legacy Migration Hazard`.
- `docs/resources/practitioner-guide/19-starting-mid-stream.md` uses backfill terminology for
  existing project artifacts.
- The 1.0 release plan decision was that there are no legacy GenDev projects to migrate.

Impact:

The guide can imply that 1.0 includes a legacy GenDev migration workflow. That conflicts with the
release decision and can confuse the distinction between imported product artifacts and legacy
methodology migration.

Remediation:

- Rename `Legacy Migration Hazard` to a term such as `Imported Authority Hazard`.
- Reserve `migration` for product, schema, deployment, or data migration work inside a project.
- Use `import`, `backfill`, or `adoption` for bringing existing non-GenDev project material under
  GenDev control.
- Cross-check the glossary and mid-stream chapter for the same terminology boundary.

### DC-007: The 1.0 Adoption Guide Is Not Integrated Into Practitioner Entry Points

Severity: Medium

Evidence:

- `docs/resources/releases/1.0.0-adoption.md` exists.
- The Practitioner Guide entry points do not consistently link to that adoption guide.

Impact:

Users starting fresh or mid-stream can miss the concise 1.0 adoption path, especially the distinction
between new-project use and existing-artifact import.

Remediation:

- Link the adoption guide from `docs/resources/practitioner-guide/README.md`.
- Link it from `04-starting-a-new-project.md` for new-project use.
- Link it from `19-starting-mid-stream.md` for existing-project import.
- Make sure the adoption guide points back to the exact Practitioner Guide chapters that implement
  the workflow.

### DC-008: Automated Documentation Coherence Checks Are Too Narrow

Severity: Medium

Evidence:

The documented validation commands passed even though the audit found:

- Active 0.5 headings in methodology role and dev-skill docs.
- Preflight omissions in the Practitioner Guide.
- Scaffold mismatches in the repository map.
- Release-state contradictions between release docs and lifecycle metadata.

Impact:

The repository can regress into visible documentation drift while still presenting a clean validation
summary.

Remediation:

- Extend `scripts/check-doc-coherence.sh` to scan all active methodology and Practitioner Guide
  documentation.
- Add targeted assertions for:
  - stale active release headings,
  - required 1.0 operator-command references,
  - scaffold path agreement,
  - release candidate versus published release wording,
  - adoption guide cross-links.
- Add regression tests for both passing and failing fixtures.

### DC-009: The 1.0 Release Plan Still Reads Like a Draft Current-State Plan

Severity: Low

Evidence:

- `docs/resources/evolution/1.0.0-production-release-plan.md` still says
  `Status: Draft plan for human review`.
- Some current-state observations in that plan describe the branch and tag state as they existed
  before the release branch implementation.

Impact:

The plan is a valid historical planning artifact, but without a post-execution status note an agent
can mistake stale observations for current repository state.

Remediation:

- Change the status to reflect that the plan was accepted for execution.
- Add a short post-execution note pointing to the execution log and current release documents.
- Clearly label initial observations as an initial planning snapshot.

### DC-010: Historical 0.5 Design Decision Records Need Clear Historical Framing

Severity: Low

Evidence:

Several design decision records under `docs/methodology/design/` still refer to 0.5 target release
work. These appear to be historical decision records, not current operating instructions.

Impact:

The content can remain intact, but active readers need a clear signal that these records are
historical and not the current release target.

Remediation:

- Add a consistent historical-record note to dated or superseded design decision records.
- Exclude explicitly historical records from stale-active-release checks.
- Keep original decision evidence intact.

## Work Packages

### DCR-WP01: Restore Release-State Honesty

Objective: Align release docs and lifecycle metadata so they accurately distinguish publication
status from release-candidate readiness.

Tasks:

- Decide whether the branch is still pre-publication or fully published.
- If pre-publication, update release index and lifecycle registry wording to candidate status.
- If publishing now, close the required gates, create the tag, and update release docs in the same
  publication commit.
- Add a check that rejects `released` lifecycle status when the planned tag does not exist.

Acceptance criteria:

- Release docs, lifecycle registry, release notes, and tags tell the same story.
- `./scripts/check-lifecycle-coherence.py --mode release` remains clean.
- A missing `v1.0.0` tag cannot coexist with active docs claiming final publication.

### DCR-WP02: Remove Stale 0.5 Active Headings

Objective: Make active methodology role and dev-skill docs release-consistent for 1.0.

Tasks:

- Replace stale active 0.5 operational-coherence headings with 1.0 or release-neutral headings.
- Preserve historical release documentation unchanged unless it is presented as active guidance.
- Add doc coherence coverage for these files.

Acceptance criteria:

- No active role or dev-skill doc contains a stale 0.5 heading.
- Historical 0.5 records remain available and clearly separate from active guidance.

### DCR-WP03: Update Practitioner Preflight for 1.0

Objective: Make the first-run and first-agent-session workflows use the 1.0 operator commands.

Tasks:

- Update `04-starting-a-new-project.md`.
- Update `16-checklists.md`.
- Update `docs/methodology/guides/start-and-next-step-protocol.md`.
- Confirm `05-working-with-the-agent.md` and `15-prompt-library.md` remain aligned with the new
  preflight language.

Acceptance criteria:

- New-project preflight requires doctor, state, and doc-coherence checks.
- First agent session guidance requires agents to report health and state evidence before execution.
- Prompt-library examples align with the updated preflight sequence.

### DCR-WP04: Reconcile the Practitioner Repository Map With the Scaffold

Objective: Make the repository map match what 1.0 actually creates and expects.

Tasks:

- Compare `03-repository-map.md` against `scripts/init-project.sh` and `docs/project-template/`.
- Remove or add `docs/project/README.md` intentionally.
- Add missing current directories and purposes.
- Consider adding a lightweight scaffold-reference check.

Acceptance criteria:

- Documented scaffold paths exist or are explicitly marked optional.
- Every first-class artifact directory is documented.
- The guide no longer describes paths that a normal initializer run does not create.

### DCR-WP05: Align Adoption, Backfill, and Migration Language

Objective: Remove accidental legacy GenDev migration implications while preserving valid imported
artifact guidance.

Tasks:

- Rename or reframe `Legacy Migration Hazard`.
- Audit the glossary, mid-stream chapter, and adoption guide for terminology consistency.
- Define the boundary between product/data migration and GenDev adoption or import.

Acceptance criteria:

- Active guidance does not imply there are legacy GenDev projects to migrate.
- Existing-artifact adoption remains clearly supported.
- Product and data migration language remains available where it refers to project-domain work.

### DCR-WP06: Integrate the 1.0 Adoption Guide Into the Practitioner Guide

Objective: Make the 1.0 adoption guide discoverable from all relevant entry points.

Tasks:

- Add cross-links from the Practitioner Guide README, new-project chapter, and mid-stream chapter.
- Add reciprocal links from the adoption guide back to the implementing Practitioner Guide chapters.
- Ensure links use stable repository-relative paths.

Acceptance criteria:

- A new practitioner can find the 1.0 adoption guide from the Practitioner Guide table of contents.
- A mid-stream practitioner can find the adoption guide from the mid-stream workflow.
- Link validation or manual path checks pass.

### DCR-WP07: Expand Documentation Coherence Automation

Objective: Make the checks catch the documentation defects found in this audit.

Tasks:

- Extend `scripts/check-doc-coherence.sh`.
- Add failing fixtures or test cases for the specific defect classes.
- Include active-doc stale release headings, preflight command references, scaffold map coverage,
  release-state wording, and adoption-guide links.

Acceptance criteria:

- The new checker fails on representative stale or missing guidance.
- The checker passes after the remediation edits.
- Release-coherence suite includes the expanded checks.

### DCR-WP08: Update Release Planning Artifacts

Objective: Prevent planning artifacts from being mistaken for current execution state.

Tasks:

- Update the 1.0 release plan status to accepted or superseded by execution.
- Add a post-execution note that points to the execution log and release-state docs.
- Mark initial current-state observations as a planning snapshot.

Acceptance criteria:

- The release plan no longer presents stale branch and tag observations as live state.
- The execution log is the obvious source for completed work and remaining gates.

### DCR-WP09: Validate and Publish the Documentation Remediation

Objective: Prove the documentation set is coherent after remediation.

Tasks:

- Run:

```text
./scripts/check-doc-coherence.sh
./scripts/check-lifecycle-coherence.py --mode candidate
./scripts/check-lifecycle-coherence.py --mode release
./scripts/test-methodology.sh --suite release-coherence
```

- If the branch remains pre-publication, document why release-mode validation is expected to pass or
  fail.
- Capture validation evidence in the execution log or remediation closeout note.

Acceptance criteria:

- All expected checks pass.
- Any deliberately deferred publication gate is explicitly recorded.
- The Practitioner Guide, methodology docs, release docs, and lifecycle metadata are mutually
  consistent.

## Recommended Execution Order

1. DCR-WP01, because release-state truth controls every other 1.0 claim.
2. DCR-WP02 through DCR-WP06, because these are user-facing documentation repairs.
3. DCR-WP07, because the checker should lock in the repaired behavior.
4. DCR-WP08, because it clarifies historical planning records after live docs are aligned.
5. DCR-WP09, because final validation should run against the complete edited set.

## Open Decisions

Recommended decisions:

- Treat the repository as pre-publication until `v1.0.0` exists and required publication gates are
  closed.
- Use release-neutral operational-coherence headings in active role and dev-skill docs unless a
  section is explicitly release-specific.
- Preserve historical 0.5 design decision records, but mark them as historical where needed.
- Use `adoption`, `import`, and `backfill` for bringing existing product artifacts under GenDev
  control; reserve `migration` for project-domain data, deployment, or schema migration work.

## Done Definition

This remediation is complete when:

- Active documentation consistently presents GenDev as the 1.0 methodology without stale 0.5 active
  headings.
- Practitioner Guide preflight, checklists, repository map, adoption paths, and prompt examples
  reflect the 1.0 operator tooling.
- Release-state documentation and lifecycle metadata agree with the actual tag and publication state.
- Automated coherence checks catch the defect classes found in this audit.
- Final validation evidence is recorded in the release execution trail.
