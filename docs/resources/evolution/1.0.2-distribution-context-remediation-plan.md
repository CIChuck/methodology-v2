# GenDev 1.0.2 Distribution Context Remediation Plan

Status: Proposed
Date: 2026-07-12
Baseline reviewed: `master` at the `v1.0.1` publication commit
Evidence sources: Briefwire dogfood project (G0 to G3, `COMMAND-LOG.md`
findings, 2026-07-12) and an independent external agent assessment of the
extracted Briefwire repository (Codex, 2026-07-12)
Target release: `1.0.2`
Planned remediation branch: `release/1.0.2-distribution-context`

## Purpose

GenDev 1.0.1 was validated exclusively inside the methodology authority
repository, the one environment where authority context and execution context
coincide. The first dogfood project (Briefwire) and the first cold external
agent assessment of an installed product repository surfaced a class of
defects that validation inside the authority repository is structurally
incapable of detecting: the distribution boundary changes the meaning of
methodology metadata and the applicability of methodology tooling, and 1.0.1
does not model that boundary at all.

This plan records the findings with their evidence, defines the work packages
that remediate them, and establishes the installed-context regression class
that prevents recurrence. The product-layer result from the same dogfood was
clean: a foreign agent correctly read project state, held the two-layer
ontology, and declined to write product code at a pending gate. The
conceptual model is sound; enforcement of the distribution boundary is what
lags.

## Scope

In scope:

- The distribution manifest, installer, and everything they place in a
  product repository.
- Context awareness of repo-scoped methodology tooling
  (`check-doc-coherence.sh`, `check-lifecycle-coherence.py`,
  `test-methodology.sh` and its suites, `gendev-doctor.sh`).
- The CI workflow rendered into product repositories.
- Template documentation gaps that cost agent round-trips during the dogfood.
- Release execution for 1.0.2 per the established publication gate procedure.

Out of scope:

- Any change to gate semantics, artifact templates beyond documentation
  fixes, or the constitution.
- The blast-radius self-reclassification policy question (recorded under
  Open Decisions).
- Briefwire itself, which remains a sample project and proceeds to G4
  independently of this plan.

## Validation Baseline

Inside the authority repository at `v1.0.1`, all checks are clean:

```text
./scripts/check-doc-coherence.sh
./scripts/check-lifecycle-coherence.py --mode release
./scripts/check-methodology.sh
./scripts/test-methodology.sh --platform-summary --suite all
```

Inside an installed, initialized product repository built exactly per the
adoption guide (install, init, close G1 to G3), the same day:

```text
./scripts/gendev-doctor.sh            clean
./scripts/project-state.sh            clean
./scripts/check-methodology.sh        clean, 0 warnings
./scripts/check-doc-coherence.sh      FAILS, exit 8, false findings
./scripts/check-lifecycle-coherence.py --mode release   FAILS, 10 errors
./scripts/test-methodology.sh --suite all               FAILS, multiple suites
```

The divergence between the two columns is the defect class this plan
remediates. The green half is the product-context toolkit; the red half is
authority-context tooling running where its assumptions are false.

## Findings

### DG-001: Distribution Context Is Not A First-Class Concept

Severity: High. The installer ships authority-context tooling and metadata
into product repositories with no marker distinguishing the installed
snapshot from live methodology authority. `check-lifecycle-coherence.py
--mode release` fails in a product repo because the shipped registry declares
a generated-contract provenance whose generator
(`generate-lifecycle-contract.py`) is deliberately not in the distribution
manifest, and pins observed published tags that exist only in the authority
repository's git history. `check-doc-coherence.sh` fails because its
tag-conditional release-identity assertions key on local tag existence, which
is meaningful only in the authority repository; an installed release index
truthfully says 1.0.1 is published while the product repo has no tag, and the
checker reports contradiction where there is none. Evidence: external
assessment findings 1 and 2.

Root cause: methodology metadata changes meaning at the installation
boundary. In the authority repository it is live state carrying coherence
obligations (is the release published, does the contract regenerate, do the
tags exist). In a product repository it is frozen provenance carrying
integrity obligations only (which version is installed, from where, is it
intact). 1.0.1 tools ask authority-context questions in a context where those
questions are meaningless.

### DG-002: Product CI Is Mis-Scoped And Red On Day One

Severity: High. The workflow rendered into product repositories runs the
authority repository's maintenance suites. Several suites copy the current
repository and assume `docs/project` is absent; in any initialized product
checkout that assumption is false by definition, so the suites fail. Any
adopter who follows the adoption guide and pushes to a remote receives a
failing CI pipeline immediately. Evidence: external assessment finding 3;
confirmed against the rendered workflow in the Briefwire repository.

### DG-003: Test Suites Assume An Uninitialized Source Repository

Severity: Medium. Independent of CI scoping, the runtime, enforcement, and
metrics suites are not executable in an initialized checkout because their
fixtures copy the source tree and then call `init-project.sh` expecting a
clean slate. Either the suites gain initialized-source tolerance or they are
scoped out of product context entirely by DG-001/DG-002 remediation; the
decision is recorded in WP-2. Evidence: external assessment finding 3,
specific line references to `tests/methodology/test-runtime-tools.sh`.

### DG-004: Failed Suites Leave Untracked Residue

Severity: Low. A failed suite run left an untracked `.tmp/` directory in the
product repository, and `.tmp/` is not covered by a shipped ignore rule.
Cleanup must run on failure paths, not only success paths. Evidence: external
assessment validation notes.

### DG-005: Provenance Pin Format Is Enforced But Undocumented

Severity: Medium. The staleness detector compares an artifact's
`Derived from` revision against the hash of the last commit touching the
source file. The template does not state the expected format; an agent's
reasonable first guess (the drafting commit's short hash) triggers a
staleness warning and costs a diagnose-and-repin round trip. The detector
behaved correctly; the documentation gap is the defect. Evidence: Briefwire
`COMMAND-LOG.md`, G2 section and finding 2.

### DG-006: Design-Interrogation Answer Form Is Enforced But Unstated

Severity: Medium. `check-methodology.sh` requires the G3 design-verification
interrogation to be recorded as labeled answer lines (`Failure modes:`,
`Scale limits:`, `Evolution risk:`, `Security boundary:`), but the
architecture template presents the interrogation as a prose prompt with no
statement of the required form. A complete prose answer fails validation
until restructured. Evidence: Briefwire `COMMAND-LOG.md`, G3 section and
finding 3.

### DG-007: No Installation Record Exists

Severity: Medium. A product repository has no durable record of which
methodology version was installed, from what source, at what commit, on what
date. The doctor infers the version from the shipped registry, and
`project.yaml` records `methodology_version`, but neither captures source
provenance or supports integrity verification against the distribution
manifest. Upgrades, audits, and support all depend on this record existing.
Evidence: implied jointly by DG-001 and the external assessment's inability
to state the installed baseline's provenance.

## Work Packages

### WP-1: Installation Record And Context Marker

The installer writes `docs/methodology/schema/installation.json` into the
target repository recording: methodology version, source repository, source
tag and commit, install date, installer options used, and the manifest
digest of every installed file. The presence of this record is the
distribution-context marker. `gendev-doctor.sh` reports context
(`authority` or `installed`), installed version, and source provenance, and
gains an integrity check that verifies installed files against the recorded
manifest digests, reporting modifications loudly. Remediates DG-001 (marker)
and DG-007 (record).

Acceptance: installing into a clean repository produces the record; the
doctor in that repository reports installed context with version and source;
deliberately modifying one installed file causes the doctor integrity check
to name it.

### WP-2: Context-Aware Authority Tooling

`check-doc-coherence.sh` and `check-lifecycle-coherence.py --mode release`
detect the installation record and exit immediately with a distinct exit
code and a one-line explanation that the check applies only to the
methodology authority repository. They are removed from the distribution
manifest in favor of not shipping questions that cannot be asked; the
context guard remains as defense in depth for copied trees. The
authority-context test suites (`test-methodology.sh` full matrix) gain the
same guard. Suite fixtures additionally gain initialized-source tolerance
where the suite remains legitimately runnable in product context, per the
DG-003 decision recorded here: suites that validate methodology internals do
not ship; the product-context suite set is defined in WP-3. Remediates
DG-001 (tooling) and DG-003.

Acceptance: in an installed repository, each guarded tool exits with the
distinct code and explanation, never with false findings; in the authority
repository, behavior is unchanged and all existing suites pass.

### WP-3: Product-Scoped CI Workflow

The installer renders a product workflow distinct from the authority
workflow. The product workflow runs what is true and useful in product
context: prerequisite installation, `gendev-doctor.sh` including the WP-1
integrity check, shell syntax over installed scripts, `check-methodology.sh`,
and the initialization-safe suite subset defined in WP-2. The authority
repository's own workflow is unchanged. Remediates DG-002.

Acceptance: a fresh install plus init plus one closed gate, pushed to a
remote, produces green CI on both runner platforms; the Briefwire repository
regenerated under 1.0.2 passes its rendered workflow.

### WP-4: Suite Hygiene

All suites clean their temporary directories on success and failure paths
(trap-based cleanup), and the installer ships an ignore rule covering the
suite scratch location. Remediates DG-004.

Acceptance: a deliberately failed suite run leaves `git status --short
--ignored` free of suite residue beyond the ignored scratch path.

### WP-5: Template Documentation Fixes And Provenance Tooling

The artifact templates state, at the point of use, the provenance revision
format (hash of the last commit touching the source file) and the
design-interrogation labeled answer form with the four required labels shown
as the answer skeleton. The practitioner guide's authoring chapters gain the
same two statements.

Beyond documentation, provenance pinning becomes a tool rather than an
incantation: `scripts/pin-provenance.sh [--check] ARTIFACT...` reads each
artifact's Derived from entries and repins every real source path to that
source's current last-touching commit hash, computed identically to the
staleness checker so the two cannot disagree. The staleness warning message
is amended to name the tool as its remedy. The tool ships in the
distribution manifest and is covered by an enforcement-tools regression case
(pin, move the source, detect staleness, repin, clean). The practitioner
runbook (shipped this release, field-validated through G3) documents the
tool as the standard step in the gate rhythm. Remediates DG-005 and DG-006.

Acceptance: an agent following only the template comments produces a
provenance pin and an interrogation block that pass `check-methodology.sh`
on first validation; doc-coherence assertions protect both template
statements; `pin-provenance.sh` resolves an induced staleness warning in one
invocation, and the warning text names the tool.

### WP-6: Installed-Context Regression Class

A new suite, runnable only in the authority repository, that builds the
installed context end to end in a scratch directory: install into an empty
repository, initialize a project, author minimal G1 evidence, close G1, then
assert the product-context contract: doctor clean with installed context
reported, `check-methodology.sh` clean, guarded authority tools exiting with
the distinct context code, product workflow commands green, no residue. This
is the regression class that makes DG-001 through DG-004 unrepeatable, and
it is the structural lesson of this release: every future release must
validate on both sides of the distribution boundary. Wired into the
authority repository's CI matrix.

Acceptance: the suite passes in the authority repository on both platforms;
reverting any single WP-1 through WP-4 change causes at least one case to
fail.

### WP-7: Release Execution 1.0.2

Version identity promoted to 1.0.2 across active surfaces per the procedure
established in 1.0.1: registry, regenerated contract, README, constitution
version line, project template, release index, release note
(`docs/resources/releases/1.0.2.md`), adoption guide update covering the
installation record and product CI, tag-conditional checker assertions keyed
to `refs/tags/v1.0.2`, and the synchronization-target patterns updated on
both sides. Publication follows the seven-gate procedure with the atomic
publication commit and tag. The 1.0.1 entry in the release index gains its
superseded-by pointer at publication.

Acceptance: the release-coherence suite passes in both tag states; the
publication commit validates only with its tag, as designed.

## Recommended Execution Order

WP-1, then WP-2 and WP-4 together, then WP-3, then WP-5, then WP-6, then
WP-7. The installed-context suite (WP-6) lands after everything it verifies
exists and before release execution, so the 1.0.2 release branch is the
first release validated on both sides of the distribution boundary.
Execution follows the cold-judge patch flow used for 1.0.1: patches built
from a fresh clone, JUDGE-README, independent review, publication gates.

## Open Decisions

- Whether an agent may self-reclassify blast radius (Briefwire reclassified
  C2 to C1 with recorded reason and owner during the dogfood). The record
  was honest; the policy question of whether classification is a
  human-reserved act belongs to the constitution and is deliberately not
  settled by this remediation. Owner: Chuck Russell.
- Whether `docs/resources` (optional install class) should ship the release
  and evolution history into product repositories at all, or a trimmed
  practitioner set. Default if undecided: unchanged for 1.0.2.

## Done Definition

1.0.2 is done when the authority repository validates clean in both tag
states, the installed-context suite passes on both platforms, a
fresh-install product repository produces green product CI on first push,
the guarded authority tools cannot emit false findings outside the authority
repository, and the seven publication gates are closed with the tag
`v1.0.2` and its GitHub Release recorded in the 1.0.2 execution log.
