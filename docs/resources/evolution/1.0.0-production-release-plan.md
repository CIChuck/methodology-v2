# GenDev 1.0 Production Release Plan

Status: Draft plan for human review
Date: 2026-07-11
Target release: `1.0.0`
Recommended tag: `v1.0.0`
Release title: `GenDev 1.0 Production Release`
Owner: TBD

## Purpose

This plan defines the work required to move GenDev from the current
`0.5.0-operational-coherence` baseline to a production-quality `1.0.0` release.
The plan is based on a deep documentation and tooling scan after the 0.5
remediation project, with special attention to coherence between the core
methodology, the practitioner guide, project scaffolding, lifecycle registry,
release notes, and validation tools.

The central 1.0 question is:

```text
What features would make this release valuable to an AI agentic developer or
practitioner who must use the methodology accurately under real delivery
pressure?
```

For 1.0, the answer must be more than "the documents are complete." The release
should give an agentic practitioner a deterministic operating surface: a clear
starting path, current-state detection, command-assisted artifact generation,
mechanical validation, release-state honesty, and enough examples to recover
from ambiguity without inventing process.

## Scope

In scope:

- update release identity from a 0.5 operational baseline to a stable 1.0
  methodology release;
- eliminate active-document contradictions about current version, tag state,
  lifecycle status, and pre-1.0 status;
- integrate 0.5 addenda into the practitioner guide's main flow;
- make the project scaffold, initializer, templates, and lifecycle registry
  tell the same story;
- improve agent-operability through command surfaces, state summaries,
  preflight checks, and richer examples;
- strengthen validation so documentation, registry, generated contracts, and
  release notes cannot silently diverge;
- define publication gates for local validation, remote CI, tag creation,
  release notes, and GitHub Release publication.

Out of scope unless explicitly added:

- rewriting the methodology's core lifecycle model from G0-G9;
- replacing current shell tooling with another implementation language;
- claiming support for project shapes that have not been tested against the 1.0
  initialization and onboarding path;
- publishing the release before the 1.0 gates in this plan are satisfied or
  explicitly waived.

## Working Assumptions

- "David and Markdown format" is interpreted as "detailed Markdown format."
  Clarify if a person named David has a required format.

## Evidence-Backed Findings

### F-01: Root README Still Presents Mixed Release Identity

Evidence:

- `README.md` declares current methodology version
  `0.5.0-operational-coherence`.
- The same section still calls the repository a "hardened pre-1.0 baseline,"
  points release notes to `docs/resources/releases/0.1.0-baseline.md`, and says
  the expected tag is `v0.1.0-baseline`.

Impact:

- An AI agent starting from the root README receives contradictory release
  authority.
- A practitioner may follow 0.1 release guidance while the lifecycle registry,
  tests, and release index target 0.5.

1.0 mitigation:

- Make the README a stable release entry point.
- Replace historical 0.1 release guidance with a current release status block.
- Move old baseline language into historical release notes only.
- For 1.0, make the README identify:
  - current version;
  - current tag;
  - latest release notes;
  - required validation command;
  - practitioner starting path;
  - publication status.

### F-02: 0.5 Release Notes And Release Index Were Not Fully Post-Publication Updated

Evidence:

- `docs/resources/releases/0.5.0-operational-coherence.md` still says
  "publication actions pending approval," "Proposed tag," and "no merge, tag,
  or push authorized by this document."
- `docs/resources/releases/README.md` lists 0.5 as released but says the tag is
  proposed and "not created pending tag approval."
- The 0.5 tag now exists at the merged release commit.

Impact:

- Release artifacts contradict actual repository state.
- A future 1.0 reviewer cannot trust the release index as a publication ledger
  without rechecking Git history.

1.0 mitigation:

- Add a pre-1.0 cleanup work package that updates 0.5 release notes and release
  index to record the observed tag, merge status, and residual CI limitation.
- Add a release-publication checklist that forces release notes and release index
  update after tag creation.
- Add a checker rule that compares the latest release index tag claim with
  `git tag --points-at HEAD` or an explicit release metadata file.

### F-03: Lifecycle Registry Is Released, But Registry README Still Describes Candidate State

Evidence:

- `docs/methodology/schema/lifecycle.json` has `registry.status: released`,
  `versions.candidate_status: released`, and
  `released_current: 0.5.0-operational-coherence`.
- `docs/methodology/schema/README.md` says `Status: Candidate for human
  ratification` and describes `registry.status: candidate`,
  `versions.candidate_status: proposed`, and `versions.released_current: null`
  as the current metadata through WP-10.

Impact:

- The human explanation of the registry conflicts with the machine-readable
  registry.
- An agent trying to reason from prose may choose candidate-mode behavior even
  after release-mode validation.

1.0 mitigation:

- Update schema README to describe the released 0.5 state and the planned 1.0
  transition.
- Preserve candidate-mode semantics as a mode description, not as the live
  current state.
- For 1.0, ensure `lifecycle.json`, generated shell contract, schema README,
  checker expectations, release notes, and root README are updated atomically.

### F-04: Published Tag Observation Is Stale In Lifecycle Registry

Evidence:

- `docs/methodology/schema/lifecycle.json` records
  `published_tags_observed: ["v0.1.0-baseline"]`.
- Current local tags include 0.1 through 0.5.
- `scripts/check-lifecycle-coherence.py` still contains an exact expectation
  for `["v0.1.0-baseline"]` in one version-consistency path.

Impact:

- The checker encodes historical observation rather than current publication
  reality.
- 1.0 release work can pass release-mode coherence while still preserving stale
  publication metadata.

1.0 mitigation:

- Decide whether `published_tags_observed` is a historical baseline observation
  or a live publication ledger.
- If live, update it and remove hard-coded 0.1-only expectations.
- If historical, rename or split the field so it cannot be mistaken for current
  release state.
- Add a current release publication ledger that can include 1.0 without
  rewriting historical observation semantics.

### F-05: Practitioner Guide Is Still Marked Draft And Pre-1.0

Evidence:

### F-07: Static Project Template And Initializer Do Not Expose The Same Scaffold

Evidence:

- `docs/project-template` visibly contains directories such as
  `architecture`, `as-built`, `build-plan`, `decisions`, `prd`,
  `security-governance`, `testing`, `traceability`, and `vision`.
- `scripts/init-project.sh` also creates `approvals`, `design`, `review`, and
  `deployment`.
- Those initializer-created directories are not visible in the tracked static
  template tree.

Impact:

- A practitioner browsing the template sees an incomplete project shape.
- An AI agent may infer the wrong canonical artifact surface before running the
  initializer.

1.0 mitigation:

- Make the static template, initializer, lifecycle registry path table,
  practitioner guide, and distribution manifest agree on one canonical scaffold.
- Add `.gitkeep` or README files for tracked empty canonical directories.
- Add a scaffold coherence test that compares the initializer-created
  directories against registry-declared canonical paths and tracked template
  directories.

### F-08: Late-Gate Artifact Creation Is Not As Ergonomic As Early Initialization

Evidence:

- `scripts/init-project.sh` renders early and phase-1 artifacts.
- Late-gate artifacts such as aggregate review, deployment approval, runbook,
  rollback, monitoring, and terminal close-out are enforced by the methodology,
  but their command-assisted creation is less obvious from the initialization
  path.

Impact:

- Agents may wait until a checker failure to discover late-gate artifacts.
- Practitioners may create late-stage files manually and drift from templates.

1.0 mitigation:

- Add command-assisted creation for all major lifecycle artifacts, not just
  project initialization and phase 1.
- Provide a single "next artifact" or "state report" command that tells the
  agent what file to create or update for the current gate.

### F-09: Test Harness Portability Needs A 1.0 Hardening Pass

Evidence:

- The aggregate suite failed in this managed sandbox because
  `tests/methodology/test-runtime-tools.sh` uses
  `$repo_root/.tmp/methodology-runtime-tools-...`.
- The suite has helper conventions around `TH_WORKDIR`, but runtime-tools still
  creates the first work root under repo-local `.tmp`.

Impact:

- A production release should validate in read-mostly or managed workspaces when
  a writable temp root is provided.
- AI agents often run inside constrained sandboxes; release validation should
  support that environment deliberately.

1.0 mitigation:

- Make test work roots consistently configurable through `TH_WORKDIR`,
  `TMPDIR`, or a documented `GENDEV_TEST_TMPDIR`.
- Add a negative portability test for a non-writable repository root with a
  writable temp directory.
- Document the expected local validation command for normal and constrained
  environments.

### F-10: Documentation Link And Path Coherence Needs A Smarter Checker

Evidence:

- A naive documentation path scan produced many apparent missing paths, mostly
  because active docs intentionally reference initialized-project paths under
  `docs/project/` that do not exist in the baseline checkout.
- Current documentation-coherence tests catch several stale-text classes, but
  they do not fully distinguish baseline-owned paths, generated initialized
  project paths, example paths, and historical release-note paths.

Impact:

- Broken real links can hide among intentional generated-project references.
- Naive link checking would create noise; no checking leaves drift undetected.

1.0 mitigation:

- Add a documentation path-reference checker with typed path classes:
  - baseline-owned file must exist in repo;
  - initialized-project path must match a registry/template pattern;
  - generated artifact path must have a command or template owner;
  - historical/provenance path may be exempt but must be under a historical
    release or provenance directory.
- Include practitioner guide and methodology guides in this checker.

### F-11: Tool-Specific Agent Guidance May Be Stale By 1.0

Evidence:

- The guide includes Codex- and Claude-specific notes.
- Agentic development tools change quickly, and 1.0 is intended to be a
  production release for AI agentic practitioners.

Impact:

- A 1.0 practitioner may rely on outdated tool-specific assumptions about
  planning mode, approvals, subagents, memory, sandboxing, or CI interaction.

1.0 mitigation:

- Refresh tool-specific chapters against current vendor behavior before 1.0.
- Add "last reviewed" dates and scope boundaries to each tool-specific chapter.
- Keep normative methodology tool-agnostic; keep tool-specific chapters
  operational and clearly subordinate to methodology authority.

### F-12: The Worked Example Is Useful But Still Thin For Production Adoption

Evidence:

- The practitioner guide states the walkthrough is intentionally thin and should
  later gain richer artifact excerpts, stronger production evidence, and more
  tool-specific addenda.
- 0.5 example tests verify current C1 and C2 examples, but the guide's main
  walkthrough still needs production-depth evidence for 1.0 confidence.

Impact:

- An AI agentic practitioner benefits most from concrete examples at ambiguity
  points: approval text, gate-log entries, remediation disposition, rollback,
  non-deployment rationale, terminal value review, and final close-out.

1.0 mitigation:

- Expand the vendor contract tracker walkthrough into a complete lifecycle path
  through G9, with concise but real-looking artifact excerpts.
- Include both deployment and explicit non-deployment terminal variants, or
  clearly choose one and document the other as a companion example.
- Add example validation commands and expected pass/fail outputs.

## 1.0 Value Opportunities For Agentic Practitioners

The following opportunities should be treated as first-class 1.0 release value,
not optional polish.

### O-01: Agent Start Contract

Add a short, deterministic "agent start contract" that tells an AI coding agent:

- which files to read first;
- how to detect whether the repository is a methodology baseline or an
  initialized project;
- how to detect the current gate;
- how to report current state to the human;
- when to ask for approval;
- when to stop rather than infer missing authority.

Candidate locations:

- `AGENTS.md`
- `README.md`
- `docs/resources/practitioner-guide/05-working-with-the-agent.md`
- generated initialized project quickstart under `docs/project/README.md`

### O-02: `doctor` Or Preflight Command

Add one command for practitioners and agents to run before serious work:

```bash
./scripts/gendev-doctor.sh
```

It should report:

- methodology version and release tag state;
- lifecycle registry and generated contract coherence;
- initialized project status;
- current gate and gate-log health, if initialized;
- required local tools;
- whether hooks are installed;
- whether remote CI status can be observed;
- next recommended validation command.

### O-03: Project State Report Command

Add a command that prints a concise project state summary:

```bash
./scripts/project-state.sh
```

It should answer the questions an agent must know before acting:

- What project slug is active?
- What gate is current?
- What artifact is authoritative for the current gate?
- What evidence is missing?
- What changed since the last approved gate event?
- What is the next allowed transition?
- Which human approval, if any, is required?

### O-04: Next Artifact Generator

Add command-assisted generation for every canonical artifact:

```bash
./scripts/new-artifact.sh --kind deployment-runbook
./scripts/new-artifact.sh --kind aggregate-review
./scripts/new-artifact.sh --kind terminal-closeout
```

This reduces the chance that an agent creates ad hoc files that satisfy prose but
miss checker fields.

### O-05: Coherence Dashboard Or Release Snapshot

Generate a release snapshot file that captures:

- release version;
- tag;
- lifecycle registry hash;
- generated contract hash;
- validation commands and results;
- remote CI URL and green result;
- release notes path;
- adoption guide path;
- known residuals.

This can be a Markdown artifact and a machine-readable JSON artifact.

### O-06: Agent-Oriented Failure Recovery

Convert common failure modes into actionable recovery flows:

- checker failed because project is uninitialized;
- gate transition missing exact evidence;
- upstream artifact changed and downstream artifacts are stale;
- phase exit attempted before independent review;
- deployment path confused with explicit non-deployment path;
- remote CI unavailable.

Each recovery flow should include:

- what the agent should inspect;
- what the agent may change;
- what requires human input;
- what command proves recovery.

### O-07: Production Adoption Path

Add a "use GenDev in an existing repo" path that combines the mid-stream guide,
artifact import rules, and first validation run into one operator-friendly
workflow.

For 1.0, this should be more important than abstract completeness: production
users often adopt methodology after code already exists.

## 1.0 Release Gates

### Gate R1: Planning Accepted

Required evidence:

- accepted 1.0 release plan;
- answered open questions or documented assumptions;
- dedicated release branch decision.

Exit criteria:

- human authorizes implementation work packages.

### Gate R2: Documentation Truthfulness Restored

Required evidence:

- 0.5 publication state corrected;
- README release-state contradictions removed;
- schema README no longer contradicts lifecycle JSON;
- practitioner guide status and version wording are accurate.

Exit criteria:

- doc coherence checker passes.

### Gate R3: Agent Operability Delivered

Required evidence:

- agent start contract;
- preflight/state command or accepted equivalent;
- command-assisted artifact creation or documented just-in-time artifact rules;
- failure recovery flows.

Exit criteria:

- a cold agent can identify current state, next artifact, and required
  validation without hidden human knowledge.

### Gate R4: Scaffold And Lifecycle Coherent

Required evidence:

- template, initializer, registry, manifest, and guide path references agree;
- scaffold coherence test passes;
- late-gate artifact creation path is tested.

Exit criteria:

- initialized project shape is mechanically reproducible and documented.

### Gate R5: Full Validation Green

Required evidence:

- local full suite pass;
- constrained-temp validation pass or documented environment requirement;
- release-mode lifecycle coherence pass;
- methodology checker pass;
- remote CI green.

Exit criteria:
