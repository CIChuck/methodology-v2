# Release 0.3.0 — Documentation Structure and Technique Neutrality

Status: Accepted
Date: 2026-06-17
Supersedes: 0.2.0-phase-loop

## Summary

Establishes Technique Neutrality as a first-order constitutional principle and
derives from it a coherent set of documentation-structure rules: canonical
artifact naming, a locked architecture-independent documentation scaffold, a
uniform Supporting Artifacts mechanism, a typed acyclic reference graph, and the
form discipline that makes supporting artifacts coherence-checkable. The method
now governs how an artifact earns authority and is named, located, and
referenced, while remaining silent on the technique that produces its content.

The maxim: the method does not specify the technique, but the technique must
blend with the method.

See docs/methodology/design/documentation-structure-decision-record.md for the
principle, the seven consequences, the rationale, and the rejected alternatives.

## Principle

Technique Neutrality (constitution keystone): GenDev governs how work earns
authority and how that authority is gated, reviewed, and kept coherent. It does
not specify how the work is conceived, modeled, or built. The blend point is the
artifact: a technique's artifacts enter through a fixed authority-and-reference
discipline, so the method can gate and coherence-check them without knowing which
technique produced them.

## Changes by consequence

Consequence 1 — Canonical artifact naming (constitution Rule 14)
- Per-project artifacts use fixed, role-based filenames identical across projects
  (vision.md, prd.md, architecture.md, traceability-matrix.md, and the rest). No
  slug or project name in any filename or cross-reference path.
- The slug survives only as a project.yaml field. Project identity is carried by
  location, the manifest, and a strictly required project: <slug> front-matter
  field on every per-project authority artifact, matching the manifest slug.
- AGENTS.md references canonical artifacts by full path; a single pointer is
  correct for all projects.
- init-project.sh: all 15 prior slug occurrences removed; canonical names
  generated; project field emitted on every authority artifact (rendered
  templates and heredoc artifacts alike).

Consequence 2 — Documentation scaffold canonical and architecture-independent
(constitution Rule 11)
- The documentation scaffold (the docs/project/ tree and canonical filenames) is
  fixed for every project regardless of stack.
- The code scaffold is architecture-determined and only recorded (via the
  technology-stack decision artifact and implementation_paths), never prescribed.

Consequence 3 — Supporting Artifacts section
- Every canonical gate-artifact template gains a uniform Supporting Artifacts
  section (15 templates), empty by default, through which technique-specific
  design artifacts attach as typed references.

Consequence 4 — Typed, acyclic reference graph (constitution Rule 12)
- Supporting-artifact references form a directed acyclic graph rooted at canonical
  artifacts: edges run from the canonical artifact to its supporters, cycles are
  forbidden, one level deep by default.
- Five-type vocabulary, each with a coherence obligation and an explicit authority
  direction: implements, satisfies, tested-by, constrained-by, refines. Provenance
  (Derived from) remains separate.

Consequence 5 — Form-disciplined supporting artifacts (constitution Rule 13)
- The method constrains form (valid kebab identifier, canonical location, required
  project field, typed relationship) and is silent on content and name. Identity
  is the typed edge plus front matter, not the filename. Textual form is
  authoritative; diagrams are embedded or referenced.

Consequence 6 — Phase-plan feature breakdown; roadmap retirement
- The phase-plan template gains a per-phase feature breakdown (Phase Detail:
  features delivered, requirements covered, dependencies, exit signal), revisable
  through the plan's Amendments section.
- phase-roadmap.md is retired: no longer generated, roadmap field removed from
  the manifest. This resolves the reconciliation deferred in 0.2.0 (the roadmap
  predated the phase plan; the phase plan now fully absorbs its role).
- init-project.sh renders the phase plan from its template rather than an inline
  heredoc, eliminating template/scaffold divergence.

Consequence 7 — Gate-log template extraction
- gate-log-template.md extracted to the methodology templates directory as the
  canonical home for all nine event-type schemas; init renders the gate log from
  it.

## Enforcement

- check-methodology.sh validates the project front-matter field (present and
  matching the manifest slug; errors on missing or mismatched) and
  supporting-artifact form (kebab filename, project field, matching slug). Tested
  on happy-path, missing, and mismatched cases for both regular and supporting
  artifacts.
- Reference-graph properties (cycle detection, typed coherence) remain normative
  but unenforced by the checker; they are the responsibility of the planned
  documentation coherence linter. This is a recorded, accepted gap.

## Documentation

- Practitioner guide updated to teach the new structure: a Canonical Naming and
  Supporting Artifacts section in the gates-and-artifacts chapter, and glossary
  entries for Canonical Naming, Reference Graph, Supporting Artifact, and
  Technique Neutrality.
- The throwaway eval-test-product sample was removed from the repository, and
  docs/project/ is gitignored so generated project output is never committed to
  the methodology repository. A companion sample is a separate, post-release
  artifact.

## Execution and review

Executed as a four-phase amendment through GenDev's own phase loop (principle and
reference rules; naming and scaffold; templates; checker and sweep). Each phase
was independently reviewed by a cold-context judge and remediated to a clean
verdict before the next began. The amendment's planning artifacts obeyed the
reference discipline the amendment introduces.
