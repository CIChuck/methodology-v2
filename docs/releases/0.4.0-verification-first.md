# Release 0.4.0 — Verification-First

Status: Accepted
Date: 2026-06-20
Supersedes: 0.3.0-documentation-structure

## Summary

Elevates verification to a first-order commitment. GenDev already held a shift-left
testing posture implicitly (the Core Principle, Rule 3, the G2 requirements gate,
the phase loop's exit tests), but it never named the commitment, never
distinguished the kinds of verification a project owes, and never bridged the gap
between a human-approved document and code an AI can be reliably tested against.

This release names the Verification-First Principle, differentiates the three
verification questions, and defines the chain by which approved documents become
reliably testable: EARS-formed acceptance criteria at G2, a human-approved
verification specification at G3, and a build loop that grades generated code
against that approved specification rather than against prose.

See docs/methodology/design/verification-first-decision-record.md for the
principle, the three questions, the verification chain, the rationale, and the
deferred follow-ons.

## Principle

Verification-First (first-order principle): a project must define how it will know
the work is correct before it builds the work, and a human must approve that
definition of correctness in a form precise enough to test. Verification is a
design input, not a downstream check.

Verification answers three exhaustive questions: behavioral (does the
implementation do what is required, including the edges?), design (does the design
hold under the conditions it must survive?), and implementation (is the code sound
and durable under change?). What unifies them is that each asks whether the work is
correct under conditions the happy path does not reveal.

Verification-First is a methodological commitment, not a technique. The method
requires the verification and gates it; it does not prescribe how it is produced.
Test-driven, behavior-driven, acceptance-test-driven, property-based, and classic
test-after development are technique, neutral under Technique Neutrality. The maxim:
define how you will know it works before you make it work.

## Changes

Phase 1 — the principle (constitution)
- Verification-First added as a first-order named principle (after the Core
  Principle), defined around the three questions.
- Rule 3 retitled from "Tests Are Design Artifacts" to "Verification Is a Design
  Input" and reframed as a consequence of the principle.

Phase 2 — EARS, the G3 verification specification, distributed criteria
- G2: acceptance criteria are expressed in EARS notation (the five templates) for
  C2/C3 projects; C1 may use plain observable criteria. Unwanted-behavior (If/then)
  criteria are required where error paths exist. Cross-cutting concerns (security,
  performance, operational, deployment) flow through the three questions, not as a
  fourth category.
- G3: the architecture gains a required, human-approved Verification Specification,
  derived from the G2 EARS criteria and approved as a faithful encoding of intent
  separately from approving code. This is the artifact that lets the build loop and
  an AI reviewer grade against certified intent rather than reinterpreted prose. A
  design-verification interrogation (failure modes, scale, evolution) is answered
  inline, proportional to blast radius.
- Phase loop: the phase build plan gains an architecture mirror check (design
  verification re-asked at phase scope against the ratified G3 architecture) and
  PRD/vision traceability on the phase exit test. The value review gains a
  scaffolded user-acceptance section (G2 scenarios executed; the objective captured,
  the subjective framed with a checklist and hints, disagreement recorded).
- All verification sections scale with blast radius; the C2 standard case is kept
  light (inline prompts, not appendices). Verification evidence attaches as
  supporting artifacts through the tested-by typed reference, not in the templates.

Phase 3 — checker and sweep
- The methodology checker validates EARS acceptance criteria for ready/accepted
  C2/C3 PRDs (C1 skipped) and the presence and human-approval of the G3 verification
  specification. The checks fail on the negative cases, not only pass the happy
  path.
- The architecture section renamed in Phase 2 was swept: all stale "acceptance
  criteria seed" references across active docs now point to the verification
  specification.

## Enforcement

The checker enforces EARS form (presence of concrete EARS criteria) and the
verification specification (presence and human approval). Per-requirement coverage
across the PRD and architecture is cross-document coherence and is assigned to the
planned coherence linter, not the checker. Verification-First gates the existence
and approval of verification criteria; it does not introduce mechanical test
execution, consistent with the Enforcement Principle.

## Deferred (recorded follow-ons, not in this release)

- Formal methods and the blast-radius-dependent rigor ladder (Design-by-Contract,
  property-based testing, model checking with TLA+/TLC, Alloy, proof assistants),
  pursued as a separate addendum. Attachment point: blast-radius scaling.
- The practitioner-guide testing chapter teaching EARS, TDD, and UAT-driven practice
  as technique.
- Project adoption (starting GenDev at a gate above G0 from existing documents),
  which depends on this release's final gate criteria.

## Execution and review

Executed as a three-phase amendment through GenDev's own phase loop. Each phase was
independently reviewed by a cold-context judge and remediated to a clean verdict
before the next began. Two phases required remediation, both for genuine findings:
the design-verification interrogation was initially a named subsection rather than
the decided inline block, and the first EARS checker matched the template's own
instructional examples and could not fail (caught by negative-case testing).
